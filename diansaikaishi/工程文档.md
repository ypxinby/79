# 工程文档

> 目的：让后续开发或 AI 接手时，能快速理解当前小车工程的系统架构、任务层设计、动作接口和扩展方式。本文档对应当前代码版本，不是远期设想。

## 1. 工程定位

这是一个 TI CCS / MSPM0G3507 嵌入式循迹小车工程。

当前主线：

```text
七路灰度循迹 + MPU6050 航向辅助 + 静态任务动作编排 + HC-SR04 前向避障停车
```

整体原则：

```text
CarController 负责底层运动控制和电机输出。
MotionAction 负责把一个任务动作转换成 CarController 的目标。
MissionManager 负责按顺序调度动作。
MissionLibrary 负责保存可选任务和正式地图任务。
menu/oled_ui 负责现场选择任务、调参和观察状态。
Ultrasonic / ObstacleMonitor / ObstacleSafety 负责前向障碍检测、状态判断和安全停车。
```

任务层不直接调用 `Motor_SetSpeed()`，所有电机输出仍统一经过 `CarController`。

## 4. 20ms 调度主线

入口在 `empty.c`。

```text
SysTick 每 100us 调用 Motor_PwmTick100us()
主循环累计到 20ms 后调用 App_Update_20ms()
```

当前 `App_Update_20ms()` 顺序：

```text
1. Imu_Update(0.02f)
2. App_UpdateHeadingObserver()
3. Ultrasonic_Update_20ms()
4. ObstacleMonitor_Update_20ms()
5. Key_Update_20ms()
6. Key_GetEvent()
7. Menu_HandleKeyEvent()
8. MissionManager_Update_20ms()
9. ObstacleSafety_Update_20ms()
10. CarController_Update_20ms()
11. 更新调试变量
12. OledUi_Update_20ms()
```

注意：

```text
MissionManager 在 CarController 前更新。
ObstacleSafety 必须在 MissionManager 后、CarController 前更新。
MissionManager/MotionAction 设置动作目标。
ObstacleSafety 根据障碍状态设置 SafetyHold / ExternalHold。
CarController 在同一个 20ms 周期内根据目标和 SafetyHold 更新实际电机输出。
```

`SysTick_Handler()` 中的 100us 周期还会执行：

```text
Motor_PwmTick100us()
Ultrasonic_Tick100us()
```

## 5. 底层运控状态机

状态定义在 `car_controller.h`：

```c
typedef enum {
    TRACK_MODE_SEEK_LINE = 0,
    TRACK_MODE_FOLLOW_LINE,
    TRACK_MODE_TURN_LEFT_90,
    TRACK_MODE_TURN_RIGHT_90,
    TRACK_MODE_LOST_RECOVER
} TrackRunMode;
```

含义：

```text
SEEK_LINE       没看到线时按目标 yaw 直线找线
FOLLOW_LINE     正常灰度循迹
TURN_LEFT_90    左 90 度单轮转弯
TURN_RIGHT_90   右 90 度单轮转弯
LOST_RECOVER    丢线后按最后方向找线
```

关键接口：

```c
void CarController_StartSeekLine(float target_yaw_deg);
void CarController_StartFollowLine(CarTurnHandlingPolicy turn_policy);
void CarController_StartTurnLeft90(void);
void CarController_StartTurnRight90(void);
void CarController_Stop(void);
void CarController_SetSafetyHold(bool enable);
bool CarController_IsSafetyHoldActive(void);
TrackRunMode CarController_GetRunMode(void);
const CarControllerFeedback *CarController_GetFeedback(void);
```

`SafetyHold` 用于避障安全停车：

```text
SafetyHold = true 时，CarController 强制电机输出为 0。
它不改变当前 run_mode，不直接完成或取消任务动作。
解除 SafetyHold 时会清理部分瞬态 PD/丢线计数，降低恢复瞬间抖动。
```

`CarControllerFeedback` 是底层给任务层的事件反馈：

```text
line_found          SEEK 找到线
line_lost           FOLLOW 出线/丢线
center_detected     中间传感器看到线
detected_turn       检测到左/右 90 度路口
turn_completed      90 度转弯完成
operation_failed    底层动作失败
```

FOLLOW 路口处理策略：

```text
CAR_TURN_POLICY_AUTO         检测到 90 度后由底层自动转弯
CAR_TURN_POLICY_REPORT_ONLY  只上报路口，任务层决定下一步
CAR_TURN_POLICY_IGNORE       忽略 90 度路口检测
```

当前任务库里的 `FOLLOW_UNTIL_LEFT_90/RIGHT_90/ANY_90` 使用 `REPORT_ONLY`，也就是检测到路口后动作完成，但不会自动转弯。是否转弯由下一个任务动作决定。

## 6. 灰度和 90 度路口检测

灰度传感器定义在 `track_sensor.h`：

```text
TRACK_SENSOR_COUNT = 7
TRACK_RAW_VALID_MASK = 0x7F
TRACK_CENTER_MASK = S3/S4/S5
TRACK_LEFT_EDGE_MASK = S1/S2
TRACK_RIGHT_EDGE_MASK = S6/S7
```

当前 90 度路口检测条件：

```text
1. 左或右边缘传感器看到黑线
2. 中间 S3/S4/S5 至少一个看到黑线
3. black_count >= TRACK_TURN_MIN_BLACK_COUNT，当前为 3
4. error 方向达到 TRACK_TURN_MIN_ABS_ERROR，当前为 200
5. 对侧边缘没有看到线
```

这样比只看边缘传感器更稳，不容易把普通大弯误判成 90 度路口。

## 7. SEEK / FOLLOW / TURN 行为

### SEEK_LINE

直线找线时，速度由搜索速度和航向修正组成：

```text
left  = search_speed + heading_correction
right = search_speed - heading_correction
```

任务层可以指定 SEEK 的目标 yaw。当前目标角计算：

```text
普通 SEEK:
target_yaw = mission_start_yaw + action_angle + YAW

第二次 SEEK:
target_yaw = mission_start_yaw + REV + YAW
```

其中：

```text
mission_start_yaw = 任务启动瞬间 Imu_GetYaw()
YAW = g_appConfig.seek_heading_offset_deg
REV = g_appConfig.second_seek_angle_deg
```

当前现场默认值：

```text
YAW = -1
REV = 215
```

### FOLLOW_LINE

灰度循迹以 PD 为主：

```text
line_correction = KP * error + KD * derivative
```

FOLLOW 动作可以根据不同结束条件完成，例如出线、检测左 90、检测右 90、运行指定时间等。

### TURN_90

90 度转弯当前继续使用现场较稳定的单轮转弯方案，不接 IMU：

```text
L90: left = 0,          right = turn_speed
R90: left = turn_speed, right = 0
```

退出条件：

```text
turn_elapsed_ms >= turn_min_ms
并且中间传感器 S3/S4/S5 重新看到黑线
```

### LOST_RECOVER

丢线恢复按最后明确方向找线，不做左右扫线：

```text
LEFT:  left = 0,             right = recover_speed
RIGHT: left = recover_speed, right = 0
NONE:  left = recover_speed, right = recover_speed
```

在 `FOLLOW_END_LINE_LOST` 任务动作里，`line_lost` 会被视为动作成功完成，用来表达“出循迹路段”。

## 8. 超声波避障安全停车

当前已完成第一版前向避障：

```text
HC-SR04 前向测距
距离滤波
障碍状态判断
SEEK / FOLLOW / LOST 中安全停车
障碍消失后恢复原任务
```

硬件引脚：

```text
HC_TRIG = GPIOA.8   输出
HC_ECHO = GPIOA.9   输入
```

注意：

```text
HC-SR04 的 ECHO 是 5V 信号，必须经过扩展板分压/电平转换后再进 MCU。
```

### Ultrasonic

文件：

```text
ultrasonic.c / ultrasonic.h
```

职责：

```text
每 100ms 请求一次测距
在 100us SysTick 中产生 TRIG 脉冲
在 100us SysTick 中非阻塞采样 ECHO 脉宽
粗略换算 distance_cm
对有效距离做 3 点中值滤波
```

当前参数：

```text
测距触发周期：100ms
TRIG 脉冲：100us 粗粒度
ECHO 超时：300 * 100us = 30ms
有效距离：3~250cm
滤波：3 次有效距离中值
```

主要接口：

```c
void Ultrasonic_Init(void);
void Ultrasonic_Tick100us(void);
void Ultrasonic_Update_20ms(void);
const UltrasonicFeedback *Ultrasonic_GetFeedback(void);
```

`UltrasonicFeedback` 中常用字段：

```text
measurement_valid   当前滤波距离是否有效
distance_cm         滤波后的距离，OLED 显示使用
raw_distance_cm     单次原始距离，调试用
echo_high           当前 ECHO 电平
state               超声波内部状态机状态
```

### ObstacleMonitor

文件：

```text
obstacle_monitor.c / obstacle_monitor.h
```

职责：

```text
读取 UltrasonicFeedback
判断前方是否 BLOCKED
提供 O:0/1 给 OLED 和 ObstacleSafety
```

当前阈值：

```text
U < 20cm  连续 3 次 -> BLOCKED，O:1
U >= 25cm 连续 3 次 -> CLEAR，O:0
20~25cm -> 保持当前状态
```

这是一段小滞回，用于避免 20cm 附近反复闪动。

主要接口：

```c
void ObstacleMonitor_Init(void);
void ObstacleMonitor_Update_20ms(void);
const ObstacleFeedback *ObstacleMonitor_GetFeedback(void);
```

### ObstacleSafety

文件：

```text
obstacle_safety.c / obstacle_safety.h
```

职责：

```text
读取 ObstacleMonitor 的 O:0/1
决定是否设置 CarController SafetyHold
决定是否设置 MissionManager ExternalHold
提供 H:0/1 给 OLED
```

当前行为：

```text
O:1 且 run_mode 为 SEEK / FOLLOW / LOST -> H:1
H:1 -> 电机强制停止，任务动作冻结
O:0 -> H:0，解除停车，任务继续
```

当前边界：

```text
L90 / R90 转弯过程中暂不自动 Hold。
这样做是为了避免转弯中停车后再恢复导致姿态不可控。
```

主要接口：

```c
void ObstacleSafety_Init(void);
void ObstacleSafety_Update_20ms(void);
bool ObstacleSafety_IsHolding(void);
```

### Hold 链路

完整链路：

```text
HC-SR04
-> Ultrasonic distance_cm
-> ObstacleMonitor O:0/1
-> ObstacleSafety H:0/1
-> CarController_SetSafetyHold()
-> MissionManager_SetExternalHold()
```

`SafetyHold` 和 `ExternalHold` 的区别：

```text
SafetyHold：底层电机强制输出 0。
ExternalHold：任务层冻结，不增加 mission_elapsed_ms/action_elapsed_ms，不切换动作。
```

调度顺序必须保持：

```text
MissionManager_Update_20ms()
ObstacleSafety_Update_20ms()
CarController_Update_20ms()
```

如果 `ObstacleSafety` 放在 `MissionManager` 前面，任务启动动作可能重置 CarController 状态，把 SafetyHold 覆盖掉。

## 9. 任务层架构

任务层由四个文件组成：

```text
motion_types.h       定义动作类型、动作参数、结果、yaw 参考系
motion_action.c      执行单个动作，判断动作是否完成
mission_manager.c    管理任务状态、动作索引、暂停/继续/取消
mission_library.c    静态任务数组和任务注册表
```

运行关系：

```text
MissionLibrary 提供任务定义
MissionManager 选择一个任务并按 action_index 取动作
MotionAction_Start() 启动当前动作
MotionAction_Update_20ms() 根据 CarControllerFeedback 判断动作结果
动作 SUCCESS 后 MissionManager 进入下一个动作
```

`timeout_ms = 0U` 表示不启用超时。当前比赛/测试任务大多不启用超时，因为目标是跑完整地图。

## 12. 如何新增地图任务

一般只改两个文件：

```text
mission_library.h
mission_library.c
```

步骤：

```text
1. 在 mission_library.h 新增任务 ID。
2. 在 mission_library.c 的 Competition missions 区新增 MotionAction 数组。
3. 在 g_missionRegistry[] 注册这个任务。
4. 编译后进入 OLED 参数页，TASK 选择到新 ID。
```

示例：

```c
#define MISSION_ID_MAP_D (13U)
```

```c
static const MotionAction g_missionMapD[] = {
    /* 先找线。 */
    ACTION_SEEK_MISSION_YAW(0.0f, 0U),
    /* 到右 90 路口，但不自动转。 */
    ACTION_FOLLOW_UNTIL_RIGHT_90(0U),
    /* 由任务层明确右转。 */
    ACTION_TURN_RIGHT_90(0U),
    /* 到终点出线后停车。 */
    ACTION_FOLLOW_UNTIL_LINE_LOST(0U),
    ACTION_STOP()
};
```

```c
{
    .mission_id = MISSION_ID_MAP_D,
    .name = "MAP-D",
    .actions = g_missionMapD,
    .action_count = ARRAY_SIZE(g_missionMapD),
    .control_profile_id = 0U
}
```

注意：

```text
FOLLOW_UNTIL_RIGHT_90 / LEFT_90 / ANY_90 只是检测路口并结束动作。
如果需要转弯，后面必须显式写 ACTION_TURN_RIGHT_90 或 ACTION_TURN_LEFT_90。
如果后面直接 ACTION_STOP，小车会在路口停车。
如果后面继续 ACTION_FOLLOW_FOREVER，小车会继续循迹。
```

## 16. IMU 和航向控制

文件：

```text
imu.c / imu.h
heading_control.c / heading_control.h
```

硬件：

```text
MPU6050
I2C 地址 0x68
SDA = GPIOA.10
SCL = GPIOA.11
独立模拟 I2C，不与 OLED 共用
```

IMU 负责：

```text
初始化 MPU6050
读取 gyro z
零偏校准
计算 corrected gyro
积分 yaw
```

HeadingControl 负责：

```text
保存 target_yaw
计算 yaw error
输出 heading_correction
```

当前航向控制主要用于 SEEK 直线找线，不用于 90 度转弯。

## 17. 不要轻易改的点

```text
不要让任务层直接调用 Motor_SetSpeed()。
不要把地图任务写进 car_controller.c。
不要把 IMU 航向控制接入 90 度转弯，当前转弯方案已现场验证较稳定。
不要恢复旧的左右扫线式 LOST_RECOVER。
不要把现场微调角 YAW 和第二次 SEEK 结构角 REV 混成一个参数。
不要让避障模块直接调用 Motor_SetSpeed()，必须通过 SafetyHold/CarController。
不要把 ObstacleSafety_Update_20ms() 放到 MissionManager_Update_20ms() 前面。
不要手改 Debug/ 下的 CCS 编译产物。
```

## 18. 后续改进方向

优先级建议：

```text
P0 继续增强任务可观测性：
   显示 detected_turn、错误码、当前动作结果等。

P1 完善 SEEK 完成策略：
   当前 SEEK 找到线后底层会切入 FOLLOW_LINE。
   后续可以做 SEEK_COMPLETE_REPORT_ONLY，让任务层完全决定找到线之后是跟线、停车还是等待。

P2 扩展正式地图任务：
   新地图优先只改 mission_library.c/h。

P3 参数持久化：
   后续可考虑 Flash 保存 TASK / SPD / YAW / REV / KP / KD / MAX。
   建议等任务框架稳定后再做。

P4 更高级运动能力：
   编码器速度闭环、按距离直行、IMU 指定角度转弯、任务 Profile、运行统计页。

P5 避障后续扩展：
   增加 OBS 观察页。
   增加舵机左右扫描。
   扫描后只显示推荐方向，暂不主动绕行。
   主动绕障需要等 TURN_TO_YAW / DRIVE_HEADING / REACQUIRE_LINE 等动作成熟后再做。
```

当前最推荐的扩展方式仍然是：

```text
先把实际地图描述成动作序列。
再只改 mission_library.c 的任务数组和注册表。
最后现场通过 OLED 选择 TASK 并测试。
```
