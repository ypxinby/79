# 工程文档

> 目的：让后续开发或 AI 接手时，能快速理解当前小车工程的系统架构、任务层接口、避障链路和扩展方式。本文档对应当前代码状态，不是远期设想。

## 1. 工程定位

这是一个 TI CCS / MSPM0G3507 嵌入式循迹小车工程。

当前主线能力：

```text
七路灰度循迹
MPU6050 yaw 辅助 SEEK / 转角 / 保持航向直行
任务动作数组编排地图
HC-SR04 前向障碍检测
FOLLOW 动作可选择 STOP_ONLY / FIXED_BYPASS
第一版固定主动绕障：停车等待 -> 转出 -> 航向直行 -> 转回 -> 回线稳定 -> 恢复原 FOLLOW
OLED 状态页 / HEAD 页 / OBS 避障页 / 参数页
PB9 舵机保持前向角度，ObstacleScanner 代码保留但默认通过功能开关关闭
```

核心分层原则：

```text
CarController       底层运动控制和电机输出
MotionAction        把一个任务动作转换成 CarController 目标
MissionManager      按顺序调度动作，处理暂停、继续、外部冻结
MissionLibrary      保存测试任务和正式地图任务，并按开发/比赛 profile 暴露不同任务列表
Ultrasonic          HC-SR04 原始测距
ObstacleMonitor     把距离转换成 O:0/1 障碍状态
ObstacleSafety      把 O:1 转换成停车保持 H:1，并处理避障期间的安全门
ObstacleAvoidance   固定主动绕障状态机，临时接管 CarController
ObstacleScanner     舵机左/右扫描模块，默认 FEATURE_OBSTACLE_SCANNER=0，不参与当前主链路
menu/oled_ui        现场调参和观察状态
```

不要让任务层、避障层直接调用 `Motor_SetSpeed()`。所有电机输出必须统一经过 `CarController`。

## 4. 100us、5ms、10ms 和 20ms 调度

`empty.c` 中：

```text
SysTick = 100us
每 100us:
  Motor_PwmTick100us()
  Gimbal_Tick100us()
  Servo_Tick100us()
  Ultrasonic_Tick100us()

每 5ms:
  Gimbal_Update5ms()

每 10ms:
  GimbalTracker_Update(0.010f)

每 20ms:
  App_Update_20ms()
```

调度分层说明：

```text
100us：底层 STEP/PWM/超声波 tick，不放复杂计算
5ms：云台 YAW/PITCH 速度斜坡和 STEP 半周期更新
10ms：GimbalTracker 视觉追踪控制计算
20ms：小车任务、按键、OLED、避障、车体控制
```

`GimbalTracker_Update()` 独立于小车 20ms 主控制，不修改 Mission、MotionAction、CarController 链路。

当前 `App_Update_20ms()` 关键顺序：

```text
1. Imu_Update()
2. App_UpdateHeadingObserver()
3. Ultrasonic_Update_20ms()
4. ObstacleMonitor_Update_20ms()
5. 如果 FEATURE_OBSTACLE_SCANNER=0，让 Servo 保持在 SV 参数角度
   如果 FEATURE_OBSTACLE_SCANNER=1，则扫描器非 active/complete 时回到 SV 参数角度
6. Key_Update_20ms()
7. Menu_HandleKeyEvent()
8. MissionManager_Update_20ms()
9. ObstacleSafety_Update_20ms()
10. ObstacleAvoidance_Update_20ms()
11. CarController_Update_20ms()
12. 更新调试变量
13. OledUi_Update_20ms()
```

当前没有调用：

```text
ObstacleScanner_Update_20ms()
```

也就是说，舵机扫描代码仍在工程中，但默认不初始化、不更新，主动避障当前走固定方向，不读取 L/R/D 推荐方向。

当前关键链路：

```text
MissionManager_Update_20ms()
ObstacleSafety_Update_20ms()
ObstacleAvoidance_Update_20ms()
CarController_Update_20ms()
OledUi_Update_20ms()
```

`ObstacleSafety` 和 `ObstacleAvoidance` 都位于 `CarController` 之前，所以最终电机输出仍统一由 `CarController` 执行。

## 5. 底层运动控制

`TrackRunMode` 定义在 `car_controller.h`：

```c
typedef enum {
    TRACK_MODE_SEEK_LINE = 0,
    TRACK_MODE_FOLLOW_LINE,
    TRACK_MODE_TURN_LEFT_90,
    TRACK_MODE_TURN_RIGHT_90,
    TRACK_MODE_TURN_TO_YAW,
    TRACK_MODE_DRIVE_HEADING,
    TRACK_MODE_LOST_RECOVER
} TrackRunMode;
```

含义：

```text
SEEK_LINE       按目标 yaw 直线找线
FOLLOW_LINE     正常灰度循迹
TURN_LEFT_90    左 90 度单轮转弯
TURN_RIGHT_90   右 90 度单轮转弯
TURN_TO_YAW     按 IMU 相对角度转向
DRIVE_HEADING   锁定当前 yaw，按时间直行
LOST_RECOVER    丢线后按最后方向恢复
```

关键接口：

```c
void CarController_StartSeekLine(float target_yaw_deg);
void CarController_StartFollowLine(CarTurnHandlingPolicy turn_policy);
void CarController_StartTurnLeft90(void);
void CarController_StartTurnRight90(void);
void CarController_StartTurnToYawRelative(float angle_deg);
void CarController_StartDriveHeading(uint32_t duration_ms);
void CarController_Stop(void);
void CarController_SetSafetyHold(bool enable);
bool CarController_IsSafetyHoldActive(void);
TrackRunMode CarController_GetRunMode(void);
const CarControllerFeedback *CarController_GetFeedback(void);
```

`SafetyHold`：

```text
true  时，CarController 强制电机输出 0
false 时，恢复当前 run_mode 继续执行
它不直接完成任务动作，也不取消任务
```

`CarControllerFeedback` 是任务层和避障层判断动作完成的主要来源：

```text
line_found          SEEK 找到线
line_lost           FOLLOW 出线/丢线
center_detected     中间灰度看到线
detected_turn       检测到左/右 90 度路口
turn_completed      90 度转弯、yaw 转向、航向直行动作完成
operation_failed    底层动作失败
```

FOLLOW 的路口策略：

```text
TURN_POLICY_AUTO         底层检测到 90 度后自动转弯
TURN_POLICY_REPORT_ONLY  只上报路口，任务层决定下一步
TURN_POLICY_IGNORE       忽略 90 度路口检测
```

当前 `FOLLOW_UNTIL_LEFT_90 / RIGHT_90 / ANY_90` 使用 `REPORT_ONLY`。检测到路口后动作完成，但不会自动转弯。是否转弯由任务数组里的下一步动作决定。

## 6. 动作语义

动作类型在 `motion_types.h`：

```c
typedef enum {
    MOTION_ACTION_SEEK_LINE = 0,
    MOTION_ACTION_FOLLOW_LINE,
    MOTION_ACTION_TURN_LEFT_90,
    MOTION_ACTION_TURN_RIGHT_90,
    MOTION_ACTION_TURN_TO_YAW,
    MOTION_ACTION_DRIVE_HEADING_TIME,
    MOTION_ACTION_REACQUIRE_LINE,
    MOTION_ACTION_WAIT,
    MOTION_ACTION_STOP
} MotionActionType;
```

### SEEK_LINE

直线找线时：

```text
left  = search_speed + heading_correction
right = search_speed - heading_correction
```

Yaw 目标参考系：

```text
ACTION_SEEK_CURRENT_YAW(yaw, timeout)
  target_yaw = current_yaw + yaw + YAW

ACTION_SEEK_MISSION_YAW(yaw, timeout)
  target_yaw = mission_start_yaw + yaw + YAW

ACTION_SEEK_SECOND_CONFIG(timeout)
  target_yaw = mission_start_yaw + REV + YAW
```

现场默认：

```text
YAW = -1
REV = 215
```

### FOLLOW_LINE

灰度循迹主要使用 PD：

```text
line_correction = KP * error + KD * derivative
```

FOLLOW 动作可以按不同条件结束：

```text
一直循迹
循迹到出线
循迹到左 90
循迹到右 90
循迹到任意 90
循迹固定时间
```

FOLLOW 动作还带避障策略：

```text
OBSTACLE_POLICY_STOP_ONLY     遇障只停车等待，障碍清除后继续
OBSTACLE_POLICY_FIXED_BYPASS  遇障后执行固定主动绕障
```

绕障方向：

```text
BYPASS_DIRECTION_RIGHT        右绕，默认
BYPASS_DIRECTION_LEFT         左绕，角度符号镜像
```

当前默认 `ACTION_FOLLOW_*` 宏都使用：

```text
OBSTACLE_POLICY_FIXED_BYPASS
BYPASS_DIRECTION_RIGHT
```

需要停车等待版本时，使用 `ACTION_FOLLOW_*_WITH_OBSTACLE(policy, direction, timeout)` 显式指定。

### TURN_90

当前 90 度转弯继续使用现场验证较稳定的单轮转弯：

```text
L90: left = 0,          right = turn_speed
R90: left = turn_speed, right = 0
```

退出条件：

```text
turn_elapsed_ms >= turn_min_ms
并且中间 S3/S4/S5 重新看到黑线
```

目前 90 度转弯没有接入 IMU yaw 闭环。

### TURN_TO_YAW

当前实现为相对当前 yaw 转动：

```c
ACTION_TURN_RELATIVE_YAW(angle_deg, timeout)
```

底层语义：

```text
target_yaw = Imu_GetYaw() + angle_deg
error = normalize(target_yaw - current_yaw) 到 -180~180 度
误差大时用 turn_speed
误差进入慢速区后降速
连续稳定一段时间后完成
超时由 MotionAction 的 timeout_ms 判断
IMU 未就绪则失败停车
```

现场已验证过：

```text
TEST-YH 可验证相对转角 + 航向直行 + 转回
当前固定右绕使用 +35 度转出、-55 度转回
```

### DRIVE_HEADING_TIME

当前实现为按时间保持当前 yaw 直行：

```c
ACTION_DRIVE_HEADING_TIME(duration_ms, timeout)
```

底层语义：

```text
进入动作时锁定当前 yaw
HeadingControl 输出左右差速修正
持续 duration_ms 后完成
超时由 MotionAction 的 timeout_ms 判断
IMU 未就绪则失败停车
```

注意：

```text
这是第一版主动绕障验证用动作，不是距离闭环。
实际位移会受电池、电机、地面摩擦和负载影响。
后续需要更稳定距离时，应新增 DRIVE_HEADING_DISTANCE，并接入编码器里程闭环。
```

### REACQUIRE_LINE

当前回线动作为两阶段：

```text
SEARCH:
  用当前 yaw 执行 SEEK_LINE
  中间传感器连续 3 个周期看到线后进入 SETTLE

SETTLE:
  切到 FOLLOW_LINE + TURN_POLICY_IGNORE
  稳定约 300ms
  期间丢线则失败
```

这样避免边缘传感器偶然扫到黑线就误判回线成功。

```text
需要主动固定绕障时，FOLLOW 动作选择 FIXED_BYPASS
```

## 8. 任务层架构

任务层四个文件：

```text
motion_types.h       动作类型、动作参数、动作结果、yaw 参考系、避障策略
motion_action.c      执行单个动作，判断动作是否完成
mission_manager.c    管理任务状态、动作下标、暂停、外部冻结
mission_library.c    静态任务数组和任务注册表
```

运行关系：

```text
MissionLibrary 提供任务定义
MissionManager 选择任务并按 action_index 取动作
MotionAction_Start() 启动当前动作
MotionAction_Update_20ms() 根据 CarControllerFeedback 判断动作结果
动作 SUCCESS 后 MissionManager 进入下一个动作
```

避障恢复接口：

```c
bool MotionAction_ReapplyControllerTarget(void);
```

作用：

```text
固定绕障临时接管 CarController 后，保留当前 action_index 和任务进度。
绕障完成时重新下发当前 FOLLOW 动作的 CarController 目标。
```

## 10. 如何新增地图任务

一般只改两个文件：

```text
mission_library.h
mission_library.c
```

步骤：

```text
1. 在 mission_library.h 新增任务 ID
2. 在 mission_library.c 的 Competition missions 区新增 MotionAction 数组
3. 在 g_missionRegistry[] 注册这个任务
4. 给正式地图动作设置合理 timeout
5. 编译后在 OLED 参数页选择新 TASK
```

示例：

```c
#define MISSION_ID_MAP_D (13U)
```

```c
static const MotionAction g_missionMapD[] = {
    ACTION_SEEK_MISSION_YAW(0.0f, MAP_SEEK_TIMEOUT_MS),
    ACTION_FOLLOW_UNTIL_RIGHT_90(MAP_FOLLOW_TIMEOUT_MS),
    ACTION_TURN_RIGHT_90(MAP_TURN_TIMEOUT_MS),
    ACTION_FOLLOW_UNTIL_LINE_LOST(MAP_FINAL_TIMEOUT_MS),
    ACTION_STOP()
};
```

正式地图默认超时常量在 `mission_library.c`：

```text
MAP_SEEK_TIMEOUT_MS    = 4000
MAP_FOLLOW_TIMEOUT_MS  = 8000
MAP_TURN_TIMEOUT_MS    = 2500
MAP_FINAL_TIMEOUT_MS   = 5000
```

如果某一段需要单独放宽，可以直接在该动作行写具体值，例如：

```c
ACTION_FOLLOW_UNTIL_RIGHT_90(12000U)
```

如果某一段 FOLLOW 遇障后只想停车，不想主动绕障：

```c
ACTION_FOLLOW_FOREVER_WITH_OBSTACLE(
    OBSTACLE_POLICY_STOP_ONLY,
    BYPASS_DIRECTION_RIGHT,
    0U)
```

如果某一段 FOLLOW 要左绕：

```c
ACTION_FOLLOW_FOREVER_WITH_OBSTACLE(
    OBSTACLE_POLICY_FIXED_BYPASS,
    BYPASS_DIRECTION_LEFT,
    0U)
```

## 11. OLED 和按键

普通页面当前循环：

```text
STATUS -> HEAD -> OBS -> STATUS
```

## 13. 超声波和 O/H 安全链路

硬件引脚：

```text
HC_TRIG = GPIOA.8   输出
HC_ECHO = GPIOA.9   输入
```

注意：HC-SR04 的 ECHO 通常是 5V 信号，必须确认扩展板已经做了分压或电平转换后再进 MCU。

### Ultrasonic

职责：

```text
每 100ms 请求一次测距
100us SysTick 中产生 TRIG 脉冲
100us SysTick 中采样 ECHO 脉宽
换算 distance_cm
对有效距离做 3 点中值滤波
```

当前参数：

```text
触发周期：100ms
TRIG 脉冲：100us 粗粒度
ECHO 超时：30ms
有效距离：3~250cm
滤波：3 次有效距离中值
```

### ObstacleMonitor

职责：

```text
读取 UltrasonicFeedback
判断前方是否 BLOCKED
提供 O:0/1 给 OLED、ObstacleSafety、ObstacleAvoidance
```

当前阈值：

```text
U < 20cm   连续 3 次 -> BLOCKED，O:1
U >= 25cm  连续 3 次 -> CLEAR，O:0
20~25cm    保持当前状态
```

另外保留一条舵机扫描保护：

```text
舵机不在中心角 SV 时，不更新 O:0/1
舵机回到中心后等待约 400ms，再允许更新 O:0/1
```

当前主循环不执行扫描器，所以一般情况下舵机会保持在 SV 中心角。

### ObstacleSafety

职责：

```text
读取 O:0/1
决定是否设置 CarController SafetyHold
决定是否设置 MissionManager ExternalHold
提供 H:0/1 给 OLED
```

普通 STOP_ONLY 行为：

```text
O:1 且 run_mode 为 SEEK / FOLLOW / LOST -> H:1
H:1 -> 电机强制停止，任务动作冻结
O:0 -> H:0，解除停车，任务继续
```

主动绕障期间行为：

```text
AVOID_STATE_WAIT_AFTER_OBSTACLE:
  SafetyHold = true
  ExternalHold = true

绕障动作执行中:
  SafetyHold = false
  ExternalHold = true
  若超声波有效距离 < 10cm，则进入紧急停车保持

避障失败:
  SafetyHold = true
  ExternalHold = true

避障完成后的 resume grace:
  短时间忽略普通 O:1，只保留 <10cm 紧急保护
```

`SafetyHold` 和 `ExternalHold` 区别：

```text
SafetyHold    底层电机强制输出 0
ExternalHold  任务层冻结，不增加任务/动作时间，不切换 action_index
```

## 14. 固定主动绕障

文件：

```text
obstacle_avoidance.c / obstacle_avoidance.h
```

启动条件：

```text
CarState == RUNNING
ObstacleMonitor_GetFeedback()->blocked == true
当前 MotionAction 是 FOLLOW_LINE，或 CarController 当前处于 FOLLOW_LINE
当前 FOLLOW 的 obstacle_policy == OBSTACLE_POLICY_FIXED_BYPASS
```

当前状态机：

```c
AVOID_STATE_IDLE = 0,
AVOID_STATE_WAIT_AFTER_OBSTACLE,
AVOID_STATE_TURN_OUT,
AVOID_STATE_DRIVE_OUT,
AVOID_STATE_TURN_TO_LINE,
AVOID_STATE_REACQUIRE_SEARCH,
AVOID_STATE_REACQUIRE_SETTLE,
AVOID_STATE_COMPLETE,
AVOID_STATE_FAILED
```

当前右绕流程：

```text
1. 发现障碍，MissionManager_SetExternalHold(true)
2. 原地停车等待 2000ms
3. TURN_OUT:        ACTION_TURN_RELATIVE_YAW(+35)
4. DRIVE_OUT:       DRIVE_HEADING_TIME(1400ms)
5. TURN_TO_LINE:    ACTION_TURN_RELATIVE_YAW(-55)
6. REACQUIRE_SEARCH: 按当前 yaw SEEK_LINE，中心连续 3 次看到线
7. REACQUIRE_SETTLE: FOLLOW_LINE + IGNORE，稳定 300ms
8. MotionAction_ReapplyControllerTarget()
9. 解除 ExternalHold，回到原 FOLLOW 动作继续任务
```

左绕时：

```text
+35 / -55 的符号镜像
```

当前实现目标不是高精度机器人控制，而是用少量可组合动作积木满足简单电赛避障：

```text
TURN_TO_YAW
DRIVE_HEADING_TIME
REACQUIRE_LINE
FOLLOW obstacle_policy
bypass_direction
任务冻结和恢复
```

功能开关在 `app_features.h`：

```c

```

```text

```

文件：

```text

```

舵机参数：

```text

中心角：OLED 参数 SV，默认 90
PWM 周期：20ms
脉宽粗略范围：0.9ms~2.1ms
```

历史扫描目标：

```text
发现障碍后停车
舵机回中并确认前方距离
舵机转左，等待稳定，记录 L
舵机转右，等待稳定，记录 R
舵机回中
OLED 显示 L / R / D
```

当前状态：

```text
FEATURE_OBSTACLE_SCANNER 默认为 0
比赛默认不调用 ObstacleScanner_Init()
App_Update_20ms() 当前不调用 ObstacleScanner_Update_20ms()
主动绕障不读取 L/R/D，也不根据扫描结果选方向
Servo 仍保持在 OLED 参数 SV 对应角度，默认 90 度前向
OBS 页默认显示超声波有效位、确认计数和距离，不显示 C/L/R/SC/D
如果要调试舵机扫描，把 FEATURE_OBSTACLE_SCANNER 改为 1 后重新编译烧录
```

保留它的原因：

```text
后续如果需要基于左右空间选择左绕/右绕，可以重新接入扫描器。
但当前针对简单避障，固定方向更省主控资源，也更容易调通正式地图。
```

## 17. 下一步建议

当前已经完成第一轮代码整理和任务组织。后续重点应转向正式地图编排和实车调参，而不是继续加复杂传感器逻辑。

推荐顺序：

```text
P1. 为正式地图新增或修改 MAP-* 任务
    先只用现有动作积木拼路线
    每段 FOLLOW 根据需要选择 STOP_ONLY 或 FIXED_BYPASS
    需要左绕的段落显式传 BYPASS_DIRECTION_LEFT
    每个正式动作都设置合理 timeout

P2. 现场微调固定绕障参数
    当前 +35 / 1400ms / -55 效果较好
    若不同场地偏差大，再考虑把 avoid_* 参数加入 OLED 菜单

P3. 等地图跑通后，再考虑扫描方向
    FEATURE_OBSTACLE_SCANNER 改为 1
    重新接入 ObstacleScanner_Update_20ms()
    增加结果新鲜度
    让 ObstacleAvoidance 根据新鲜 L/R/D 选择方向

P4. 最后再做 DRIVE_HEADING_DISTANCE
    需要编码器里程闭环
    适合比赛版稳定性提升，不适合现在抢第一版落地
```

当前最重要的边界：

```text
任务负责策略
MotionAction 负责动作语义
ObstacleAvoidance 负责临时接管和恢复
ObstacleSafety 负责停车和紧急安全门
CarController 负责唯一电机输出
ObstacleScanner 当前保留但不参与主动绕障
```

## 18. 不要轻易改的点

