# 工程文档

> 目的：让后续开发或 AI 接手时，能快速理解当前小车工程的系统架构、任务层接口、避障链路和扩展方式。本文档对应当前代码版本，不是远期设想。

## 1. 工程定位

这是一个 TI CCS / MSPM0G3507 嵌入式循迹小车工程。

当前主线能力：

```text
七路灰度循迹
MPU6050 yaw 辅助直线找线
任务动作数组编排地图
HC-SR04 前向避障停车
PB9 舵机扫描左右距离并显示推荐方向
OLED 现场调参和状态观察
```

核心分层原则：

```text
CarController      底层运动控制和电机输出
MotionAction       把一个任务动作转换成 CarController 目标
MissionManager     按顺序调度动作，处理暂停、继续、外部冻结
MissionLibrary     保存测试任务和正式地图任务
Ultrasonic         HC-SR04 原始测距
ObstacleMonitor    把距离转换成 O:0/1 障碍状态
ObstacleSafety     把 O:1 转换成停车保持 H:1
ObstacleScanner    停车后舵机左/右扫描并给出 L/R/D
menu/oled_ui       现场调参和观察状态
```

不要让任务层、避障层直接调用 `Motor_SetSpeed()`。所有电机输出必须统一经过 `CarController`。

## 4. 20ms 和 100us 调度

`empty.c` 中：

```text
SysTick = 100us
每 100us:
  Motor_PwmTick100us()
  Servo_Tick100us()
  Ultrasonic_Tick100us()

每 20ms:
  App_Update_20ms()
```

当前 `App_Update_20ms()` 关键顺序：

```text
1. Imu_Update()
2. App_UpdateHeadingObserver()
3. Ultrasonic_Update_20ms()
4. ObstacleMonitor_Update_20ms()
5. 非扫描时，让 Servo 回到 SV 参数角度
6. Key_Update_20ms()
7. Menu_HandleKeyEvent()
8. MissionManager_Update_20ms()
9. ObstacleSafety_Update_20ms()
10. ObstacleScanner_Update_20ms()
11. CarController_Update_20ms()
12. 更新调试变量
13. OledUi_Update_20ms()
```

重要时序：

```text
MissionManager_Update_20ms()
ObstacleSafety_Update_20ms()
ObstacleScanner_Update_20ms()
CarController_Update_20ms()
```

`ObstacleSafety` 必须在 `MissionManager` 之后、`CarController` 之前。这样任务层先给出本周期目标，避障再决定是否强制停车，最后底层运动控制统一输出电机。

## 5. 底层运动控制

`TrackRunMode` 定义在 `car_controller.h`：

```c
typedef enum {
    TRACK_MODE_SEEK_LINE = 0,
    TRACK_MODE_FOLLOW_LINE,
    TRACK_MODE_TURN_LEFT_90,
    TRACK_MODE_TURN_RIGHT_90,
    TRACK_MODE_LOST_RECOVER
} TrackRunMode;
```

含义：

```text
SEEK_LINE       按目标 yaw 直线找线
FOLLOW_LINE     正常灰度循迹
TURN_LEFT_90    左 90 度单轮转弯
TURN_RIGHT_90   右 90 度单轮转弯
LOST_RECOVER    丢线后按最后方向恢复
```

关键接口：

```c
void CarController_StartSeekLine(float target_yaw_deg);
void CarController_StartFollowLine(CarTurnHandlingPolicy turn_policy);
void CarController_StartTurnLeft90(void);
void CarController_StartTurnRight90(void);
void CarController_Stop(void);
void CarController_SetSafetyHold(bool enable);
bool CarController_IsSafetyHoldActive(void);
TrackRunMode CarController_GetRunMode(void);
const CarControllerFeedback *CarController_GetFeedback(void);
```

`SafetyHold`：

```text
true  时，CarController 强制电机输出 0
false 时，恢复原 run_mode 继续执行
它不直接完成任务动作，也不取消任务
```

`CarControllerFeedback` 是任务层判断动作完成的主要来源：

```text
line_found          SEEK 找到线
line_lost           FOLLOW 出线/丢线
center_detected     中间灰度看到线
detected_turn       检测到左/右 90 度路口
turn_completed      90 度转弯完成
operation_failed    底层动作失败
```

FOLLOW 的路口策略：

```text
TURN_POLICY_AUTO         底层检测到 90 度后自动转弯
TURN_POLICY_REPORT_ONLY  只上报路口，任务层决定下一步
TURN_POLICY_IGNORE       忽略 90 度路口检测
```

当前 `FOLLOW_UNTIL_LEFT_90 / RIGHT_90 / ANY_90` 使用 `REPORT_ONLY`。检测到路口后动作完成，但不会自动转弯。是否转弯由任务数组里的下一步动作决定。

## 6. 灰度和 90 度判断

七路灰度定义在 `track_sensor.h`。

当前 90 度路口判断大致要求：

```text
1. 左或右边缘传感器看到黑线
2. 中间 S3/S4/S5 至少一个看到黑线
3. black_count >= TRACK_TURN_MIN_BLACK_COUNT，当前为 3
4. error 方向达到 TRACK_TURN_MIN_ABS_ERROR，当前为 200
5. 对侧边缘没有看到线
```

这样比只看边缘传感器更稳，不容易把普通大弯误判成 90 度路口。

## 7. SEEK / FOLLOW / TURN 行为

### SEEK_LINE

直线找线时：

```text
left  = search_speed + heading_correction
right = search_speed - heading_correction
```

Yaw 目标参考系：

```text
普通 SEEK:
target_yaw = mission_start_yaw + action_angle + YAW

第二次 SEEK:
target_yaw = mission_start_yaw + REV + YAW
```

当前现场默认：

```text
YAW = -1
REV = 215
```

`YAW` 用于小角度修正，`REV` 用于第二次反向找线角度。

### FOLLOW_LINE

灰度循迹主要使用 PD：

```text
line_correction = KP * error + KD * derivative
```

FOLLOW 动作可以按不同条件结束：

```text
一直循迹
循迹到出线
循迹到左 90
循迹到右 90
循迹到任意 90
循迹固定时间
```

### TURN_90

当前 90 度转弯继续使用现场验证较稳定的单轮转弯：

```text
L90: left = 0,          right = turn_speed
R90: left = turn_speed, right = 0
```

退出条件：

```text
turn_elapsed_ms >= turn_min_ms
并且中间 S3/S4/S5 重新看到黑线
```

目前 90 度转弯没有接入 IMU yaw 闭环。

## 8. 任务层架构

任务层四个文件：

```text
motion_types.h       动作类型、动作参数、动作结果、yaw 参考系
motion_action.c      执行单个动作，判断动作是否完成
mission_manager.c    管理任务状态、动作下标、暂停、外部冻结
mission_library.c    静态任务数组和任务注册表
```

运行关系：

```text
MissionLibrary 提供任务定义
MissionManager 选择任务并按 action_index 取动作
MotionAction_Start() 启动当前动作
MotionAction_Update_20ms() 根据 CarControllerFeedback 判断动作结果
动作 SUCCESS 后 MissionManager 进入下一个动作
```

`timeout_ms = 0U` 表示不启用超时。当前地图相关动作大多不启用超时，因为目标是跑完整地图。

## 9. 动作接口

动作类型在 `motion_types.h`：

```c
typedef enum {
    MOTION_ACTION_SEEK_LINE = 0,
    MOTION_ACTION_FOLLOW_LINE,
    MOTION_ACTION_TURN_LEFT_90,
    MOTION_ACTION_TURN_RIGHT_90,
    MOTION_ACTION_WAIT,
    MOTION_ACTION_STOP
} MotionActionType;
```

常用动作宏在 `mission_library.h`：

```text
ACTION_SEEK_CURRENT_YAW(yaw_deg, timeout)
ACTION_SEEK_MISSION_YAW(yaw_deg, timeout)
ACTION_SEEK_SECOND_CONFIG(timeout)

ACTION_FOLLOW_FOREVER(timeout)
ACTION_FOLLOW_FOR_TIME(duration, timeout)
ACTION_FOLLOW_UNTIL_LINE_LOST(timeout)
ACTION_FOLLOW_UNTIL_LEFT_90(timeout)
ACTION_FOLLOW_UNTIL_RIGHT_90(timeout)
ACTION_FOLLOW_UNTIL_ANY_90(timeout)

ACTION_TURN_LEFT_90(timeout)
ACTION_TURN_RIGHT_90(timeout)
ACTION_WAIT_MS(wait_time_ms)
ACTION_STOP()
```

注意：

```text
FOLLOW_UNTIL_RIGHT_90 只检测右 90 并结束动作，不会自动右转
需要右转时，下一步必须显式写 ACTION_TURN_RIGHT_90(0U)
需要停车时，下一步写 ACTION_STOP()
需要继续循迹时，下一步写 ACTION_FOLLOW_FOREVER(0U)
```

## 11. 如何新增地图任务

一般只改两个文件：

```text
mission_library.h
mission_library.c
```

步骤：

```text
1. 在 mission_library.h 新增任务 ID
2. 在 mission_library.c 的 Competition missions 区新增 MotionAction 数组
3. 在 g_missionRegistry[] 注册这个任务
4. 编译后在 OLED 参数页选择新 TASK
```

示例：

```c
#define MISSION_ID_MAP_D (13U)
```

```c
static const MotionAction g_missionMapD[] = {
    /* 先按任务起点 yaw 找第一段线。 */
    ACTION_SEEK_MISSION_YAW(0.0f, 0U),
    /* 循迹到右 90 路口，只上报，不自动转。 */
    ACTION_FOLLOW_UNTIL_RIGHT_90(0U),
    /* 任务层明确选择右转。 */
    ACTION_TURN_RIGHT_90(0U),
    /* 继续循迹到出线，然后停车。 */
    ACTION_FOLLOW_UNTIL_LINE_LOST(0U),
    ACTION_STOP()
};
```

注册：

```c
{
    .mission_id = MISSION_ID_MAP_D,
    .name = "MAP-D",
    .actions = g_missionMapD,
    .action_count = ARRAY_SIZE(g_missionMapD),
    .control_profile_id = 0U
}
```


## 14. 超声波避障

硬件引脚：

```text
HC_TRIG = GPIOA.8   输出
HC_ECHO = GPIOA.9   输入
```

注意：HC-SR04 的 ECHO 通常是 5V 信号，必须确认扩展板已经做了分压或电平转换后再进 MCU。

### Ultrasonic

文件：

```text
ultrasonic.c / ultrasonic.h
```

职责：

```text
每 100ms 请求一次测距
100us SysTick 中产生 TRIG 脉冲
100us SysTick 中采样 ECHO 脉宽
换算 distance_cm
对有效距离做 3 点中值滤波
```

当前参数：

```text
触发周期：100ms
TRIG 脉冲：100us 粗粒度
ECHO 超时：30ms
有效距离：3~250cm
滤波：3 次有效距离中值
```

接口：

```c
void Ultrasonic_Init(void);
void Ultrasonic_Tick100us(void);
void Ultrasonic_Update_20ms(void);
void Ultrasonic_ResetFilter(void);
const UltrasonicFeedback *Ultrasonic_GetFeedback(void);
```

`Ultrasonic_ResetFilter()` 当前用于舵机换方向后清空旧方向的滤波残留，避免左/右扫描读到上一个方向的距离。

### ObstacleMonitor

文件：

```text
obstacle_monitor.c / obstacle_monitor.h
```

职责：

```text
读取 UltrasonicFeedback
判断前方是否 BLOCKED
提供 O:0/1 给 OLED 和 ObstacleSafety
```

当前阈值：

```text
U < 20cm   连续 3 次 -> BLOCKED，O:1
U >= 25cm  连续 3 次 -> CLEAR，O:0
20~25cm    保持当前状态
```

另外有一条扫描保护：

```text
舵机不在中心角 SV 时，不更新 O:0/1
舵机回到中心后等待约 400ms，再允许更新 O:0/1
```

原因：舵机扫左/右时，超声波看到的是侧向距离，不能用来清除前方障碍。

### ObstacleSafety

文件：

```text
obstacle_safety.c / obstacle_safety.h
```

职责：

```text
读取 O:0/1
决定是否设置 CarController SafetyHold
决定是否设置 MissionManager ExternalHold
提供 H:0/1 给 OLED
```

当前行为：

```text
O:1 且 run_mode 为 SEEK / FOLLOW / LOST -> H:1
H:1 -> 电机强制停止，任务动作冻结
O:0 -> H:0，解除停车，任务继续
```

当前边界：

```text
TURN_LEFT_90 / TURN_RIGHT_90 过程中暂不自动 Hold
```

这是为了避免转弯中停车后再恢复导致姿态不可控。

完整链路：

```text
HC-SR04
-> Ultrasonic distance_cm
-> ObstacleMonitor O:0/1
-> ObstacleSafety H:0/1
-> CarController_SetSafetyHold()
-> MissionManager_SetExternalHold()
```

`SafetyHold` 和 `ExternalHold` 区别：

```text
SafetyHold    底层电机强制输出 0
ExternalHold  任务层冻结，不增加任务/动作时间，不切换动作
```

## 15. 舵机扫描避障

硬件引脚：

```text
SERVO = GPIOB.9
```

文件：

```text
servo.c / servo.h
obstacle_scanner.c / obstacle_scanner.h
```

舵机参数：

```text
角度范围：35~145 度
中心角：OLED 参数 SV，默认 90
PWM 周期：20ms
脉宽粗略范围：0.9ms~2.1ms
```

当前扫描目标不是主动绕障，而是：

```text
发现障碍后停车
舵机回中并确认前方距离
舵机转左，等待稳定，记录 L
舵机转右，等待稳定，记录 R
舵机回中
OLED 显示 L / R / D
```

当前扫描角：

```text
CENTER = SV 参数
LEFT   = 145 度
RIGHT  = 35 度
```

当前等待和滤波策略：

```text
每次换方向后调用 Ultrasonic_ResetFilter()
每个方向等待约 600ms 后采样
```

这样做是为了解决一个实际问题：

```text
如果舵机扫到侧面较远距离，ObstacleMonitor 可能误以为前方障碍消失。
随后 H:1 被释放，小车往前蠕动一小段，又再次检测到障碍并重复扫描。
```

当前修正：

```text
ObstacleMonitor 只在舵机中心稳定后才允许更新 O:0/1
ObstacleScanner 换方向时清空超声波滤波
扫描等待时间从约 300ms 增加到约 600ms
```

## 17. 后续扩展建议

### P0：先稳定停车扫描

目标：

```text
O:1 后必须稳定 H:1
扫描过程中小车不能往前蠕动
L/R/D 显示稳定可信
SV 中心角现场可校正
```

可继续改进：

```text
增加扫描状态显示，例如 SC:C/L/R/DONE
把 LEFT/RIGHT 扫描角做成可调参数
增加中心距离 C 显示，用于确认前方障碍是否仍在
```

### P1：把扫描结果做成任务可用的事件

目前 `ObstacleScanner` 只显示推荐方向。下一步可以新增接口：

```c
bool ObstacleScanner_IsComplete(void);
ObstacleDirection ObstacleScanner_GetRecommendedDirection(void);
```

然后让任务层或新的避障策略层读取结果。

建议先不要直接让 `ObstacleScanner` 调电机。

### P2：增加主动绕障前的底层动作

主动绕障需要更可靠的动作原语，建议先补：

```text
TURN_TO_YAW(angle)        按 IMU yaw 转到指定角
DRIVE_HEADING(ms/yaw)     按 yaw 直行一段时间
DRIVE_UNTIL_CLEAR         前方距离大于阈值后结束
REACQUIRE_LINE            绕障后重新找线
```

这些动作成熟后，再把主动绕障写成任务序列，而不是在避障模块里硬写电机控制。

### P3：任务库继续扩展地图

新地图仍建议只改：

```text
mission_library.h
mission_library.c
```

优先把路线描述成动作数组：

```text
SEEK
FOLLOW_UNTIL_RIGHT_90
TURN_RIGHT_90
FOLLOW_UNTIL_LEFT_90
TURN_LEFT_90
FOLLOW_UNTIL_LINE_LOST
STOP
```

这样底层稳定性和地图策略可以分开调。

### P4：参数持久化

等现场参数基本稳定后，再考虑 Flash 保存：

```text
TASK
SPD
YAW
REV
KP
KD
MAX
SV
```

现在先不做持久化，避免把调试阶段的错误参数固化进去。

### P5：更强的可观测性

可以继续增强 OLED：

```text
扫描状态页：C/L/R/DONE
动作结果页：last_action_result / error_code
底层反馈页：line_found / line_lost / detected_turn
避障页：raw U / filtered U / O / H / L / R / D
```

## 18. 不要轻易改的点

```text
不要让任务层直接调用 Motor_SetSpeed()
不要把地图任务写进 car_controller.c
不要让避障模块直接控制电机
不要把 ObstacleSafety_Update_20ms() 放到 MissionManager_Update_20ms() 前面
不要用舵机侧向距离直接清除前方障碍状态

不要手改 Debug/ 下的 CCS 编译产物
```

## 19. 当前最推荐的开发节奏

后续如果继续做比赛地图：

```text
1. 先描述路线
2. 只在 mission_library.c 写动作数组
3. OLED 选择 TASK 测试
4. 如果动作原语不够，再扩展 motion_action / car_controller
```

后续如果继续做避障：

```text
1. 先确认 O/H/U/L/R/D 显示稳定
2. 再增加扫描状态显示
3. 再抽象“绕障动作”
4. 最后才做自动绕障和回线
```

当前工程最重要的边界是：

```text
任务负责策略
CarController 负责运动
ObstacleSafety 负责停车冻结
ObstacleScanner 负责观察，不直接驾驶
```
