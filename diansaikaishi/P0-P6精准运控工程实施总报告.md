# 2. 硬件资源与信号

## 2.1 主控和底盘

- 主控：TI MSPM0G3507，工程由CCS/SysConfig管理。
- 驱动：TB6612类双路直流电机驱动接口。
- Motor A在当前软件语义中作为左轮，Motor B作为右轮。
- 电机业务命令统一为`-1000..1000` normalized command；闭环开启后该值不是PWM本身，而是目标轮速映射输入。

| 左轮方向 | PA13 AIN1、PA12 AIN2 | 正命令时AIN1=1、AIN2=0 |

## 2.3 七路循迹信号

当前`TRACK_BLACK_LEVEL=1`，GPIO高电平表示检测到黑线。`TrackSensor_ReadRaw()`把S1～S7依次映射到bit0～bit6，Runtime中置1表示该路看到黑线。

当前权重：

```text
S1   S2   S3  S4  S5  S6  S7
-300 -200 -100  0 +100 +200 +300
```

多路同时有效时使用有效传感器权重的算术平均。S1/S7还承担P5方向记忆：S1单侧有效记录LEFT，S7单侧有效记录RIGHT，为后续全白急转提供方向。

# 3. 系统调度

## 3.1 真实调度链

```text
10 kHz SysTick（100 us）
  ├─ 软件PWM回退tick（仅FEATURE_HW_MOTOR_PWM=0）
  ├─ 云台STEP时序
  ├─ 舵机PWM时序
  └─ 超声波时序

每1 ms
  ├─ 系统毫秒时间累加
  ├─ heartbeat/watchdog ISR检查
  └─ 生成5 ms、10 ms、20 ms pending

20 ms pending
  → 主循环原子取走全部积压计数
  → pending>1时，其余周期记为drop，不连续补跑旧控制周期
  → elapsed_ms = 本次开始时间 - 上次开始时间
  → elapsed_ms最大钳位100 ms
  → App_Update_20ms(elapsed_ms)
  → 完成后提交heartbeat
```

设计原因是：控制系统更需要“当前真实状态+真实经过时间”，而不是在主循环恢复后连续执行多个已经过期的20 ms控制决策。积压仍通过miss/drop/overrun/dt clamp计数保留诊断。

## 3.2 App_Update_20ms当前顺序

当前非轮速测试模式的真实顺序为：

```text
WheelSpeedEstimator_Update(elapsed_ms)
→ WatchdogMonitor应用待处理故障
→ Imu_Update(elapsed_ms)
→ HeadingObserver（当前ENABLE_HEADING_CONTROL=1，因此独立observer分支不执行）
→ Ultrasonic_Update_20ms
→ ObstacleMonitor_Update_20ms
→ 舵机/扫描器维护
→ Key_Update_20ms + Menu按键处理
→ 再次检查watchdog/急停；命中则立即停止并提前返回
→ MissionManager_Update_20ms
→ ObstacleSafety_Update_20ms
→ ObstacleAvoidance_Update_20ms
→ CarController_Update_20ms
→ MotorControl_Update(elapsed_ms)
→ RuntimeSnapshot/调试遥测/OLED更新
```

关键点：

- 编码器delta由`WheelSpeedEstimator_Update()`每周期唯一读取和清零。
- Mission/MotionAction选择本周期run_mode和目标；CarController在单一switch中执行该mode。
- P4开启时CarController只向MotorControl提交normalized目标，MotorControl最后调用`Motor_SetSpeed()`。
- OLED逻辑约每100 ms刷新一次，不参与控制决策。

## 3.3 其他周期

- 5 ms：云台`Gimbal_Update5ms()`。
- 10 ms：视觉Yaw/Pitch跟踪器和`GimbalTracker_Update(0.010f)`。
- 100 us：云台STEP、舵机、超声波，以及软件PWM回退时的PWM GPIO更新。
- 20 kHz硬件PWM：由TIMG8外设自主运行，不依赖20 ms主循环或100 us软件翻转。

# 4. 总体控制架构

## 4.1 正常控制数据流

```text
MissionLibrary任务数组
        ↓
MissionManager：动作推进、重试、暂停、外部冻结
        ↓
MotionAction：把任务动作转换为CarController启动请求
        ↓
CarController单一run_mode
        ├─ SEEK_LINE
        ├─ FOLLOW_LINE → P5 LineController
        ├─ TURN_LEFT/RIGHT_90（旧显式动作）
        ├─ TURN_TO_YAW
        ├─ DRIVE_HEADING
        └─ LOST_RECOVER（旧路径）
        ↓
左右normalized命令（-1000..1000）
        ↓
MotorControl_SetNormalizedTarget
        ↓
左右独立PI + 前馈 + 目标斜坡 + 安全检查
        ↓
Motor_SetSpeed
        ↓
TIMG8 20 kHz PWM + TB6612方向GPIO
        ↓
左右电机
        ↓
编码器ENCA/ENCB
        ↓
WheelSpeedEstimator滤波速度反馈到MotorControl
```

## 4.2 控制权约束

- 同一20 ms周期只有一个`run_mode`由CarController的switch执行正常运动目标。
- P5、DRIVE_HEADING、TURN_TO_YAW不是并行控制器，而是互斥run_mode。
- 避障需要临时接管时，MissionManager被`external_hold`冻结，由ObstacleAvoidance显式启动TURN/DRIVE/SEEK/FOLLOW模式。
- 急停、watchdog和SafetyHold在正常run_mode分派前阻断输出。
- 业务层不直接写PWM；P4开启时正常业务最终统一经过MotorControl。
- 只有`FEATURE_WHEEL_SPEED_CONTROL=0`时，CarController保留的开环分支才直接调用`Motor_SetSpeed()`。

# 5. P0～P1安全与工程基础

## 5.1 20 ms积压、真实elapsed和快照

P0～P1把原先不可观测的20 ms due改成pending计数，增加：

- `app_20ms_missed_count`：新周期到来时旧pending仍未消费；
- `app_20ms_drop_count`：积压合并后没有逐周期补跑的数量；
- `app_20ms_overrun_count`：一次20 ms更新执行超过20 ms；
- `app_dt_clamp_count`：真实elapsed超过100 ms并被钳位；
- `app_last_elapsed_ms`：最近一次控制elapsed。

`RuntimeSnapshot`汇总任务、CarState、run_mode、故障、安全、编码器、P4、P5、IMU和P6.2诊断字段。

## 5.2 软件急停和复位

按键以20 ms更新；`KEY_LONG_TICKS=40`，长按约800 ms。

```text
K2长按
→ EmergencyStop_Trigger
→ Motor_Stop立即强制低
→ SafetyHold=true
→ MissionManager_ReportExternalFailure
→ Fault=ESTP
→ CAR_STATE_ERROR
```

急停活动期间，主循环每周期再次执行`EmergencyStop_Enforce()`。K3长按调用`EmergencyStop_Reset()`，清急停、watchdog、绕障、MotorControl、CarController、Mission和Fault，但不会重新初始化IMU，也不会重新校准bias；复位后保持停车和READY，旧目标不会自动恢复。

非急停状态下K3长按也执行故障、watchdog、绕障、CarController、Mission和MotorControl复位。

## 5.3 heartbeat/watchdog

当前是**软件heartbeat监视**，不是硬件WWDT：

- 每次完整20 ms控制周期完成后提交heartbeat。
- SysTick 1 ms上下文计算heartbeat年龄。
- 超过100 ms时，ISR直接`Motor_Stop()`，即使主循环卡死也先停车。
- 主循环恢复后应用`HBT` Fault、SafetyHold和任务失败。
- `FEATURE_HARDWARE_WATCHDOG=0`，`empty.syscfg`没有WWDT资源，不得宣称具备硬件自动复位覆盖。

## 5.4 故障与OLED简码

当前主要Fault简码：

| Fault | OLED简码 | 当前用途 |
|---|---|---|
| TURN_TO_YAW超时 | `YTO` | 转角控制器内部真实elapsed达到timeout |
| TURN_TO_YAW IMU无效 | `YIMU` | 运行中IMU失效 |
| 避障各阶段失败 | `AOUT/ADRV/ALIN/ASRC/ASET/AINV` | 锁存失败阶段、detail/context |
| heartbeat超时 | `HBT` | 软件watchdog超时 |
| 软件急停 | `ESTP` | K2长按，优先保持该Fault |
| CarController非法mode | `CTRL` | switch default安全停车 |
| MotorControl故障 | `MCTL` | 配置、dt、反馈、目标刷新等闭环错误 |

# 6. P2硬件PWM

## 6.1 当前实现

`FEATURE_HW_MOTOR_PWM=1`时：

- TIMG8时钟32 MHz；
- period count为1600；
- PWM频率20 kHz；
- CCP0/PB15驱动左轮PWM；
- CCP1/PB16驱动右轮PWM；
- duty由normalized绝对值映射到0～1599 compare；
- 速度符号通过AIN/BIN方向GPIO表达。

方向改变时先强制对应PWM通道为低，再切方向引脚，最后恢复PWM，避免带占空切向。`Motor_Stop()`在关中断的原子区间内把两个PWM通道强制低、方向引脚清零、Runtime输出清零，因此可从ISR安全路径调用。

`FEATURE_HW_MOTOR_PWM=0`时，PB15/PB16改为GPIO，由100 us SysTick和100级计数形成约100 Hz软件PWM。该路径没有删除，仍是明确回退方式。

# 7. P3编码器与轮速估算

## 7.1 解码和符号

左右ENCA均配置为上升沿+下降沿中断。每个ENCA边沿到来时读取ENCA和ENCB：

```text
phaseA == phaseB → 原始计数 +1
phaseA != phaseB → 原始计数 -1
```

这是ENCA双边沿×2解码，不是完整AB四边沿×4。`encoder_ppr_x2=734`的精确定义是：**在当前ENCA双边沿解码方式下，车轮旋转一整圈得到的每轮脉冲数**。

20 ms观测层原子关中断读取并清零左右`int32_t` delta，同时读取并清除溢出标志。方向修正为：

```text
left_delta  = raw_motor_a_delta * (-1)
right_delta = raw_motor_b_delta * (+1)
```

因此当前工程约定前进时左右delta和速度均为正，后退时均为负。

## 7.2 几何和公式

```text
encoder_ppr_x2 = 734 pulse/rev
wheel_diameter_cm = 6.5 cm
wheel_track_cm = 11.25 cm
distance_per_pulse = pi * 6.5 / 734 cm
```

速度直接由本周期signed pulse和真实elapsed计算；里程直接累计signed pulse，不从滤波速度反推。滤波速度采用alpha=0.35的一阶IIR。

参数非法、dt非法、编码器溢出或累计溢出都会进入`error_flags`。`valid`要求错误为0且非stale；单次elapsed超过60 ms会标记stale。P4开启后，估算器invalid/stale/error会被MotorControl提升为闭环故障并停车。

## 7.3 实车验收

已确认：

- 前进LD/RD/LS/RS均为正；
- 后退均为负；
- 停车后delta和原始速度归零；

当前累计距离只用于观测和未来动作基础，尚未用于停车、转向或任务结束条件。

# 8. P4双轮速度闭环

## 8.1 开关和调用链

```text
FEATURE_WHEEL_SPEED_CONTROL=0
CarController → Motor_SetSpeed（旧开环路径）

FEATURE_WHEEL_SPEED_CONTROL=1
CarController → MotorControl_SetNormalizedTarget
              → MotorControl_Update
              → Motor_SetSpeed
```

当前开关为1；普通业务的闭环电机出口是MotorControl。`FEATURE_WHEEL_SPEED_TEST=0`，固定轮速测试入口已关闭，正常Mission链恢复。

## 8.2 目标映射和PI

```text
target_speed_cmps = normalized_command / 1000 * 60.0
error = ramped_target_speed - measured_filtered_speed
feedforward = ramped_target / 60.0 * 1000 * 0.5
P = kp_used * error
I = Ki * integral(error)
output = feedforward + P + I
```

左右独立参数当前相同：

| 参数 | 左轮 | 右轮 |
|---|---:|---:|
| `Kp_accel` | 4.0 | 4.0 |
| `Kp_overspeed` | 5.0 | 5.0 |
| `Ki` | 1.0 | 1.0 |
| 前馈增益 | 0.5 | 0.5 |

其他参数：积分限幅±300、积分释放倍率4、最大加速度60 cm/s²、最大减速度120 cm/s²、目标刷新超时100 ms。

## 8.3 anti-windup和快速稳定

- 输出饱和且误差继续推动同方向饱和时，拒绝候选积分。
- 实测速度越过目标、误差换向且旧积分仍推动原方向时，使用4倍积分释放速度。
- 超速侧使用Kp=5，正常加速侧维持Kp=4，避免为了压超调而增加启动冲击。
- 正目标输出限制为0～1000，负目标限制为-1000～0；超速时允许快速降到0，但不反向电制动。
- 目标换向时先按减速度斜坡回到0，清积分，再允许反向。
- 单轮目标为0时立即清该轮控制状态；双轮均为0时直接`Motor_Stop()`。

## 8.4 故障行为

以下条件会停止输出，其中配置、dt、估算器、反馈范围和目标刷新错误会锁存并上报`MCTL`：

- 配置非法；
- `elapsed_ms==0`或大于100 ms；
- 编码器估算器invalid、stale或error；
- 反馈超出±1000 cm/s合理范围；
- 非零目标超过100 ms未刷新；
- 急停或watchdog；
- SafetyHold或CarState不是RUNNING。

急停复位和K3完整复位都会调用`MotorControl_Reset()`，旧目标、积分和PWM不会自动恢复。

- 稳态PWM约250，与30 cm/s时前馈值250一致；
- 无持续振荡，MCTRL V/S/E正常。

P4正式冻结。P5和P6只提交normalized目标，不得绕过P4直接输出PWM。

# 9. P5稳定循迹链路

## 9.1 当前正式链路

```text
S1～S7数字黑线输入
→ 有效传感器权重平均raw_error
→ error IIR（alpha=0.35）
→ 基于真实elapsed的微分
→ derivative IIR（alpha=0.20）
→ correction = 0.35 * filtered_error
             + 0.010 * filtered_derivative
→ correction限幅±120
→ left=220+correction，right=220-correction
→ 极低非零命令归零
→ P4双轮速度闭环
→ TIMG8硬件PWM
```

当前没有P5内部速度调度，也没有P5目标变化率限制；执行层变化率统一由P4目标斜坡承担。

## 9.2 参数与转向符号

```text
base = 220
correction ∈ [-120, 120]
普通FOLLOW左右命令范围 = 100..340
对应P4目标轮速 = 6.0..20.4 cm/s
```

- 线偏左：误差为负，correction为负，左轮变慢、右轮变快，车辆向左修正。
- 线偏右：误差为正，correction为正，左轮变快、右轮变慢，车辆向右修正。
- 普通FOLLOW不允许负轮目标。
- `0<command<50`时直接归零，避免极低目标附近反复启停；当前正常PD范围最低为100，该门槛主要保护边界配置。

## 9.3 P5.2A三步全白恢复

状态仅包括：

```text
FOLLOW
LOST_TURN_LEFT
LOST_TURN_RIGHT
STOP
```

### 第一步：FOLLOW记录方向

- S1有效且S7无效：`turn_mark=LEFT`。
- S7有效且S1无效：`turn_mark=RIGHT`。
- S1、S7同时有效：不更新方向。
- 中间传感器不单独建立新方向；在普通FOLLOW期间原可靠标记会保留到全白。
- 建立标记时仍然运行正常滤波PD，不立即进入固定转向。

### 第二步：七路全白立即急转

```text
全白 + LEFT标记  → left=0，right=280
全白 + RIGHT标记 → left=280，right=0
全白 + 无有效标记 → STOP，stop_reason=NO_DIRECTION
```

全白第一帧立即进入LOST_TURN，不做2～3帧确认。新P5路径不会进入旧`TRACK_MODE_LOST_RECOVER`，旧90°自动转向也不会在普通V2 FOLLOW中抢占；旧90°检测只在任务明确使用REPORT_ONLY时作为事件上报。

### 第三步：重新见到任意黑线

LOST_TURN中只要`active_count>0`：

1. 退出固定0/280急转；
2. 清除`lost_elapsed_ms`、timeout和stop reason；
3. 清除旧turn mark；
4. 清除PD滤波/微分历史，使首帧derivative为0；
5. 当前帧立即重新执行S1/S7方向标记；
6. 当前帧立即按P5.1滤波PD输出。

边界序列：

```text
S1 → 全白 → S1 → 全白
```

重新见到S1的当前帧会立即重新建立LEFT，因此下一帧再次全白仍进入LOST_TURN_LEFT，不会因清除旧标记而进入NO_DIRECTION。

若重新见到的只有S2～S6，由于退出LOST_TURN时先清除旧标记，当前帧不会建立新标记，避免把上一个弯道方向带到新的全白区。

## 9.4 搜索超时与任务语义

当前源码的`line_lost_search_max_ms`为**2000 ms**。达到上限后：

- LineController进入STOP；
- 左右目标持续0/0；
- `recover_timeout=1`；
- `stop_reason=RECOVER_TIMEOUT`；
- LineController本身不设置`CAR_STATE_ERROR`。

无方向全白同样只在LineController内部锁存STOP。上层FOLLOW任务是否结束取决于动作的结束条件；`TEST-BASIC`使用FOLLOW_FOREVER，因此会保持停车但不会因为P5 STOP自动变成全局故障。

## 9.5 实车结论与冻结

实车结果：

- 普通直线和普通曲线平滑；
- 左右修正方向正确；
- `TEST-BASIC`可以持续循迹，不再因动作timeout自动停车；
- 当前大弯全白急转恢复链稳定，满足现阶段电赛需求；
- P4闭环、硬件PWM和急停链没有退化。

P5正式冻结。当前不再增加REACQUIRE状态、动态速度调度、弯道/直角分类、复杂宽线处理或新的P5目标斜坡。

# 10. P6.1 MPU6050与Yaw链路

## 10.1 原始问题与排查结论

P6.1之前，IMU链路存在以下工程问题：

- `imu.c`同时包含软件I2C、寄存器配置、校准、数据处理和积分，诊断字段不足；
- 上层曾固定调用`Imu_Update(0.02f)`，不能反映真实20 ms调度抖动；
- I2C读取失败时直接漏掉该段角度；
- Yaw无限累计，只有部分控制误差做局部归一化；
- MPU6050曾以错误朝向安装，纠正安装后Yaw符号反转；
- 一度出现真实旋转90°但Yaw只显示约40°。

实车排查首先确认量程和角速度换算：缓慢9秒转90°时，OLED `A10≈100`，即偏置后gyro-Z约10°/s，说明raw到dps基本正确。随后通过积分计数、累计dt和angle increment诊断定位并修正积分链，最终真实90°能够显示约90°。

## 10.2 安装和方向约定

- MPU6050板面与底盘平行；
- 板面正面朝上；
- Z轴垂直地面；
- 安装翻转会导致Z轴和Yaw符号反转；
- 统一只在角速度进入积分前乘`imu_yaw_axis_sign`；
- 当前`imu_yaw_axis_sign=-1`；
- 工程统一约定：**顺时针Yaw为正，逆时针Yaw为负**。

HeadingControl、TURN_TO_YAW和误差计算中不再额外取负来补偿安装方向。

## 10.3 量程、灵敏度和寄存器回读

当前初始化明确写入并回读：

```text
GYRO_CONFIG = 0x08
FS_SEL = (GYRO_CONFIG >> 3) & 0x03 = 1
量程 = ±500°/s
灵敏度 = 65.5 LSB/(°/s)
```

灵敏度由回读FS_SEL动态选择。虽然代码保留250/500/1000/2000°/s四档换算表，但当前要求FS_SEL必须为1；回读失败或不是1时初始化失败，避免寄存器量程与固定灵敏度不一致。

其他配置：`SMPLRT_DIV=0x07`，内部1 kHz分频到125 Hz；`CONFIG=0x03`，DLPF约44 Hz。明显超过490°/s的数据不参与积分。

## 10.4 零偏和1.5 dps死区

上电初始化后调用：

```c
Imu_CalibrateGyroBias(200U);
```

校准前丢弃50帧，再采集200个样本；有效样本少于90%则校准失败。校准要求车辆完全静止。

每帧处理为：

```text
gyro_before_bias = raw / sensitivity
gyro_after_bias = gyro_before_bias - bias
若 abs(gyro_after_bias) < 1.5°/s，则置0
```

1.5°/s死区用于抑制静止噪声长期积分漂移。代价是极慢真实旋转可能被忽略，但对电赛小车正常转向速度影响很小。

## 10.5 真实dt和短缺帧

当前接口：

```c
void Imu_Update(uint32_t elapsed_ms);
```

真实elapsed由调度层传入，IMU内部转换为秒。当前正常成功样本dt范围是**5～100 ms**。dt异常时仍保留读数和诊断，但不进行危险积分，并清除积分历史。

短缺帧策略：

1. 保存上一帧有效、已扣bias/死区/axis sign后的gyro-Z；
2. 累计距上次有效样本的真实时间；
3. 1～2个20 ms周期的读取失败可暂时保持valid，并标记`short_gap_compensating`；
4. 下一次成功读取且跨区间不超过60 ms时，用上一有效值与当前值做梯形积分；
5. 缺失区间达到60 ms时清除历史，不再跨区补偿；
6. 距上次成功读取达到100 ms时`stale=1、valid=0`；
7. stale或历史失效后的恢复首帧只重建历史，不一次性补偿整个缺口。

注意：100 ms是正常控制dt上限和stale阈值；60 ms只用于I2C缺帧桥接上限，二者不能混写。

## 10.6 最终积分与归一化

```text
raw counts
→ 根据回读FS_SEL选择LSB/(°/s)
→ 换算gyro-Z dps
→ 减去开机bias
→ 1.5 dps死区
→ 乘imu_yaw_axis_sign(-1)
→ 0.5 * (previous_gyro + current_gyro) * real_dt_s
→ yaw += angle_increment
→ Angle_Normalize180(yaw)
```

唯一公共归一化函数在`angle_utils.c/.h`，输出范围：

```text
[-180°, 180°)
```

`Imu_ResetYaw()`只清Yaw、累计积分诊断和积分历史，不清已完成的零偏校准。普通任务切换、急停和SafetyHold也不重新校准bias。

## 10.7 实车验证状态

已明确验证：

- 正确水平安装后静止Yaw漂移很小；
- 顺时针为正、逆时针为负；
- 真实旋转90°时Yaw约90°；
- 量程、灵敏度、真实dt和积分链比例正确。

`Angle_Normalize180()`的±180°范围由源码确认，但现有记录没有单独给出跨越±180°的量化实车数据，因此不把“大角度转向精度”写成P6.1已验收；该项属于P6.2后续测试。

## 11.2 HeadingControl和DRIVE_HEADING

动作启动时锁存当前Yaw：

```text
drive_heading_target_yaw = Imu_GetYaw()
```

每周期公式：

```text
yaw_error = Angle_Normalize180(target_yaw - current_yaw)
derivative = -gyro_z_dps
correction = clamp((20 * yaw_error + 5 * derivative) / 10,
                   -30, +30)
left_command  = search_speed(100) + correction
right_command = search_speed(100) - correction
```

实测`-gyro_z`作为微分反馈得以保留：当车辆正在朝误差减少方向旋转时，微分项抑制继续加大输出。

DRIVE_HEADING左右命令范围理论上为70～130，不会因为航向保持直接变成原地转向。目标经P4转换为约4.2～7.8 cm/s。

HeadingControl接收真实`elapsed_ms/1000`，dt要求`0<dt<=0.100 s`。当前公式不对dt做误差差分，因为微分直接来自陀螺仪；dt字段用于有效性和诊断。

若IMU在动作启动前无效，`MotionAction_Start()`立即失败并停车；运行中无效时CarController输出0/0并进入ERROR。P5 V2 FOLLOW每周期重置HeadingControl，不会与DRIVE_HEADING同时输出。

## 11.3 TURN_TO_YAW目标和方向

相对转角目标：

```text
target_yaw = Angle_Normalize180(current_yaw + relative_angle)
error = Angle_Normalize180(target_yaw - current_yaw)
```

当前工程Yaw顺时针为正。`YAW_TURN_REVERSE_DIRECTION=1`使正误差输出左轮正、右轮0，即向右/顺时针转；负误差输出左轮0、右轮正，即向左/逆时针转。该输出方向仍需P6.2实车确认。

## 11.4 FAST/SLOW/SETTLE状态机

```text
             abs(error) <= 12°
TURN_FAST ----------------------> TURN_SLOW
  | 左/右外轮命令130               | 外轮命令65
  |                                |
  |                                | abs(error) <= 5°
  |                                v
  |                           TURN_SETTLE
  |                           左右输出0/0
  |                                |
  |     abs(error) > 7°             | abs(error)<=5°
  +<-------------------------------+ 且abs(gyro)<=5°/s
                                   | 连续100ms
                                   v
                                  DONE
```

当前参数：

- FAST命令：`turn_speed=130`；
- SLOW命令：`max(turn_speed/2, min_slow_command)=max(65,60)=65`；
- FAST→SLOW：12°；
- 进入SETTLE：5°；
- SETTLE退出回SLOW：7°；
- 停稳角速度：5°/s；
- 连续稳定时间：100 ms。

SETTLE期间持续0/0，不通过反向PWM主动制动。迟滞5°/7°用于避免SLOW和SETTLE在边界反复切换。

## 11.5 TURN_TO_YAW超时

CarController内部用真实elapsed累计`turn_elapsed_ms`。动作显式提供timeout时使用动作值；为0时使用集中默认4000 ms。

达到timeout：

```text
左右目标0/0
→ heading_action_result=TIMEOUT
→ error_code=YAW_TURN_TIMEOUT
→ Fault=YTO
→ CAR_STATE_ERROR
→ MotionAction收到TIMEOUT
→ Mission不在ERROR状态下自动重试
```

开发任务`TEST-YH`的两次±40°转角显式timeout均为2000 ms，因此不能用该任务直接声称验证了默认4000 ms路径。

# 14. 安全优先级和故障语义

## 14.1 实际优先级

```text
软件急停 / watchdog / 真实闭环故障
> SafetyHold
> 当前显式TURN_TO_YAW run_mode
> 当前DRIVE_HEADING run_mode
> 当前P5 FOLLOW_LINE run_mode
> SEEK、旧TURN_90等普通动作
```

TURN、DRIVE和P5不是靠数值优先级同时竞争，而是由单一`run_mode`互斥。急停/watchdog在App层提前返回；SafetyHold在CarController分派前拦截；MotorControl还会独立检查安全和反馈。

## 14.2 各类异常当前行为

| 异常 | 立即输出 | 是否进入CAR_STATE_ERROR | 是否Fault锁存 | 上层结果 |
|---|---|---|---|---|
| K2长按急停 | `Motor_Stop()` | 是 | `ESTP` | Mission外部失败 |

# 16. 当前冻结基线

已明确冻结：

- P2硬件PWM频率、通道和方向路径；
- P3 PPR、轮径、轮距和方向修正；
- P4双轮速度闭环、PI、前馈、斜坡、anti-windup和快速整定；
- P5滤波PD参数和全白保向急转恢复；
- P6.1量程、灵敏度、bias、死区、axis sign、真实dt和积分公式。

后续原则：

1. 不为解决P6.2问题顺手调整P4/P5参数；
2. P5/P6不绕过MotorControl直接输出PWM；
3. 不再给Yaw增加经验比例或在控制器中额外取负；
4. 不增加不必要的复杂状态或姿态融合；
5. P6.2先定位启动ERR，再验证方向和状态机，最后才允许小幅调参；
6. P6.2验收后再开始距离闭环。

# 17. 已知边界和风险

1. MPU6050没有磁力计，Yaw是相对航向，长时间运行仍可能缓慢漂移。
2. 1.5°/s死区会忽略极慢真实旋转。
3. P5方向记忆依赖进入全白前S1或S7实际扫到黑线；没有方向时会锁存停车。
4. P5全白恢复使用固定0/280急转，不分类普通大弯、锐角、直角或路口。
5. P5搜索超时后当前不会自动进入全局ERROR；FOLLOW_FOREVER可能保持任务RUN但车辆STOP。
6. P6.2当前存在启动无动作并ERR的未定位问题，不能直接开始Kp/Kd或转速整定。
7. P6.2左右90°、大角度对称性和最终停稳精度尚未验证。
8. 精确180°在`[-180,180)`归一化边界存在方向二义性；需要任务显式约定方向，初测可用±179°。
9. DRIVE_HEADING使用时间完成，不是距离完成；地面和负载会影响实际行驶距离。
10. `wheel_track_cm`和累计里程已存在，但当前没有距离闭环动作。
11. OLED是只读的，但目前直接读取多个模块Runtime，并未统一只读RuntimeSnapshot。
12. 硬件WWDT尚未分配；当前watchdog只能在SysTick仍运行时提供软件heartbeat停车。

# 18. 下一步准确起点

> **当前准确起点**：旧`TEST-YH`启动ERR不再作为当前测试入口。先架空运行`TASK VAL=3 TEST-TURN`，确认+90°方向、角度和超时；再运行`TASK VAL=4 TEST-HEAD`，确认目标Yaw为开机零点且左右纠偏方向正确。两个任务不得与FOLLOW、旧TURN_90或LOST_RECOVER混测。

下一次工作不应从调参开始，而应从P6.2启动故障复现和证据采集开始：

1. 保持车辆架空或置于安全空旷区域；
2. 上电静止完成IMU校准；
3. STATUS确认`HOME T3 READY`、无Fault且`IMU:1`；
4. IMU页记录`I/C/V/S/A/CF`；
5. HEAD启动前记录当前Yaw、目标Yaw和IMU Ready；
6. K2短按运行`TASK VAL=3 TEST-TURN`，验证+90°相对转角；
7. 运行`TASK VAL=4 TEST-HEAD`，验证朝开机Yaw 0°定向前进；
8. 分别记录左右命令、纠偏方向、动作完成和安全停车，不再混入循迹状态；
9. P6.2通过后冻结；
10. 最后实施编码器距离闭环动作，并与已验证Yaw保持组合。

在P6.2通过前，不开始距离闭环，不修改P4/P5冻结参数。
