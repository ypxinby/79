# 云台调试记录

## 2026-07-16 P1 单轴冒烟测试

### 问题现象

PITCH 轴接线后，云台步进电机最初没有反应。

### 最终确认的正确接线

```text
MSPM0 拓展板 PB4  -> D36A ST1
MSPM0 拓展板 PA1  -> D36A DIR1
MSPM0 拓展板 PA2  -> D36A EN1
MSPM0 拓展板 GND  -> D36A GND
MSPM0 拓展板 5V   -> D36A 5V
```

### 根因

1. 原方案误用了 `PA3` 作为 PITCH STEP，但当前天猛星 2026 小车拓展板没有引出 `PA3`。
2. `PA3` 在 MSPM0G3507 上属于 LFXIN 低频晶振相关引脚，应避开，不作为云台 GPIO/PWM 使用。
3. D36A 驱动板 `EN1` 为高电平工作、低电平休眠，而测试代码最初按低有效使能处理，导致驱动没有被使能。

### 修正

1. PITCH STEP 从 `PA3` 改到 `PB4`，接 D36A `ST1`。
2. PITCH DIR 保持 `PA1`，接 D36A `DIR1`。
3. PITCH EN 保持 `PA2`，接 D36A `EN1`。
4. 将测试代码中的使能极性改为高有效：

```c
#define GIMBAL_EN_ACTIVE_LOW (0)
```

### 验证结果

重新编译下载后，PITCH 步进电机已经可以转动。说明当前 P1 的基础 STEP/DIR/EN 链路可用。

## 2026-07-16 P6 5ms 云台控制入口

### 修改内容

1. 保留 `Gimbal_Tick100us()` 作为 STEP 细脉冲入口。
2. 新增 `Gimbal_Update5ms()` 作为后续实时追踪控制入口。
3. `empty.c` 中由 100us SysTick 派生 5ms pending 计数，主循环先处理云台 5ms，再处理原有 20ms 小车任务。
4. 5ms pending 上限为 2，避免 OLED 或其他 20ms 任务阻塞后补太多历史控制 tick。

### 当前验证方式

OLED `GIMBAL` 页面新增 `T5` 显示。进入页面后，`T5` 应持续增长，表示 5ms 云台控制入口正在被主循环服务。

### 当前边界

`Gimbal_Update5ms()` 目前只作为控制心跳和后续闭环入口，不执行 OLED、UART、软件 I2C，也不改变现有小车任务链顺序。

## 2026-07-16 P7 Stop/Hold/Release 状态收拢

### 修改内容

1. 公开云台接口新增：

```text
Gimbal_StopHold()
Gimbal_Release()
```

2. 云台反馈新增三态模式：

```text
REL: Released，驱动释放，EN 关闭
HLD: Holding，驱动使能但不运动
MOV: Moving，正在执行相对角度运动
```

3. OLED `GIMBAL` 页面第 4 行显示：

```text
M: 模式
E: EN 使能状态
T5: 5ms 控制心跳
```

### 按键定义

```text
K1 短按：切换到下一个页面
K1 长按：进入参数菜单，保持原工程操作习惯
K2 短按：PITCH +5°
K3 短按：Release，释放电机
K2 长按：StopHold，停止并保持
K3 长按：Release，释放电机
```

### 验收现象

```text
上电默认 REL，电机可被手动轻轻转动
K2 长按后变 HLD，电机有保持力
K2 短按发运动命令后变 MOV，运动结束自动回 HLD
运动中 K2 长按应立即停止并保持
HLD 状态下 K3 短按或长按应变 REL，电机释放
```

### 后续注意

1. 后续双轴引脚分配必须以天猛星 2026 小车拓展板实际引出引脚为准，不只看 MSPM0G3507 芯片封装能力。
2. 避开 `PA3/PA4` 这类低频晶振相关引脚。
3. D36A 控制逻辑按当前实测记录处理：`ST` 上升沿有效，`DIR` 高低电平切换方向，`EN` 高电平工作。

## 2026-07-16 P7 复测异常：疑似下载旧固件

### 现象

```text
OLED 未显示预期的 M:REL E:0 状态行。
K2 长按仍表现为大角度转动。
K3 按下仍表现为反方向小角度转动。
```

### 判断

源码中的 GIMBAL 页按键映射已经是：

```text
K2 短按：PITCH +5°
K2 长按：StopHold
K3 短按/长按：Release
```

而上述异常现象与旧版 OLED 小角度测试映射高度一致：

```text
K2 长按：PITCH +30°
K3 短按：PITCH -5°
```

同时当前 `Debug/diansaikaishi.out` 产物中仍能找到旧 OLED 字符串 `GIMBAL PITCH` 与 `K2:+5/H K3:REL`，说明调试板上很可能运行的仍是旧构建产物。

### 修正

1. GIMBAL 页新增明显版本戳：`GIMBAL P7R2`。
2. GIMBAL 页固定显示当前按键事件：`K:<event>`。
3. 下一轮验证先确认 OLED 第一行是否为 `GIMBAL P7R2`。若不是，应先在 CCS 中 Clean/Rebuild 并确认下载的是当前工程生成的 `diansaikaishi.out`。

### 复测结果

```text
已重新烧录正确代码后，K3 现在可以正常释放电机。
```

### 结论

P7 阶段的异常主要来自旧固件/旧产物下载，不是 Release 逻辑本身错误。当前 Release 链路已通过验证。

## 2026-07-17 P9 当前角度与软限位

### 修改内容

1. 云台 PITCH 轴新增开环软件角度状态：

```text
target_deg_x10：目标角度 x10
completed_deg_x10：当前估算角度 x10
min_limit_deg_x10：最小软限位
max_limit_deg_x10：最大软限位
limit_clamped：目标是否被软限位裁剪
```

2. 当前 PITCH 软限位：

```text
最小角度：-90.0°
最大角度：+90.0°
```

3. 新增绝对目标角接口：

```text
Gimbal_PitchMoveToDeg(float target_deg)
```

4. `Gimbal_PitchMoveRelativeDeg()` 内部改为：

```text
当前估算角度 + 相对角度 -> 目标角度 -> 软限位裁剪 -> 实际相对运动
```

5. OLED GIMBAL 页版本标记更新为 `GIMBAL P9 L:x`，其中 `L:1` 表示最近一次目标角被软限位裁剪。

### 当前边界说明

当前角度是开环估算值，没有编码器或零位开关校准。上电时默认软件角度为 0°。`Release` 只关闭电机使能，不代表真实机械位置回零；如果释放后手动转动电机，软件角度会与真实角度产生偏差。

### 验收方式

```text
OLED 第 0 行显示：GIMBAL P9 L:0
K2 短按一次，目标角和当前角向 +5° 附近变化
连续短按 K2，目标角最大应被限制在 +90°，不会继续无限增加
触发上限后 OLED 第 0 行应出现 L:1
K2 长按仍应 StopHold
K3 短按/长按仍应 Release
```

## 2026-07-17 P10 云台水平轴命名迁移

### 背景

后续目标追踪逻辑需要利用小车 IMU 记录的车身航向角 `car_yaw_deg` 做反向补偿，使云台在小车大角度转弯时仍保持指向目标方向。因此当前这一路步进电机在业务语义上应视为水平 YAW 轴，而不是 PITCH 轴。

### 修改内容

1. 新增 YAW 轴主接口：

```text
Gimbal_YawInit()
Gimbal_YawTick100us()
Gimbal_YawUpdate5ms()
Gimbal_YawMoveToDeg()
Gimbal_YawMoveRelativeDeg()
Gimbal_YawStopHold()
Gimbal_YawRelease()
Gimbal_YawGetFeedback()
```

2. `app.c / empty.c / menu.c / oled_ui.c` 已切换到 `Gimbal_Yaw*` 接口。
3. 旧 `Gimbal_Pitch*` 命名不再作为 YAW 兼容包装使用，保留给后续第二个 PITCH 电机。
4. 更早的无轴名 `Gimbal_*` 接口暂时保留为单轴测试兼容包装，内部转发到 `Gimbal_Yaw*`。
5. `gimbal.c` 内部状态变量从 pitch 语义迁移为 yaw 语义：

```text
g_yawCommandStartDegX10
g_yawCurrentDegX10
g_yawTargetDegX10
g_yawLimitClamped
```

6. OLED GIMBAL 页版本标记更新为：

```text
GYAW P10 L:x
```

### 当前边界

P10 只做命名和接口语义迁移，不引入 IMU 补偿闭环。P9 的 `-90° ~ +90°` 临时软限位仍保留，用于当前阶段安全验收；后续做世界方向锁定或 360°追踪时，需要继续把角度范围模式化。

当前工程按二维云台预留命名：

```text
Gimbal_Yaw*：当前已接入的水平旋转轴
Gimbal_Pitch*：后续第二个俯仰电机预留，当前源码不占用
```

### 后续目标关系

后续世界方向锁定可按以下关系实现：

```text
gimbal_target_yaw_deg = target_world_yaw_deg - car_yaw_deg
```

其中：

```text
car_yaw_deg：小车自身航向角，来自 IMU
target_world_yaw_deg：目标在世界坐标里的方向
gimbal_target_yaw_deg：云台相对车身的目标角度
```

### 验收方式

```text
OLED 第 0 行显示：GYAW P10 L:0
K2 短按：YAW +5°
K2 长按：StopHold
K3 短按/长按：Release
角度显示、软限位和 P9 行为保持一致
```

## 2026-07-17 P11 按运控文档建立 YAW 轴位置模型

### 修改背景

前面 P8~P10 主要用于打通单轴 STEP/DIR/EN、OLED 按键验收和接口命名。P11 开始按 `云台运控逻辑.md` 的底层模型收拢，不再把固定步数测试逻辑当作正式运控核心。

### 修改内容

1. YAW 轴新增有效 STEP 上升沿累计：

```text
estimated_steps += direction
```

2. `GimbalStepperTestFeedback` 新增：

```text
int64_t estimated_steps
```

3. `gimbal.c` 高层 YAW 状态改为从累计步数反算：

```text
continuous_deg_x10 = estimated_steps * 3600 / 3200
wrapped_deg_x10    = continuous_deg_x10 mod 3600
turn_count         = floor(continuous_deg_x10 / 3600)
```

4. 相对运动命令改为连续目标角累加：

```text
Gimbal_YawMoveRelativeDeg(delta)
=> target_continuous_deg += delta
```

5. 去掉 P9 正式逻辑中的 `-90° ~ +90°` 裁剪依赖。当前 P11 不再用 `limit_clamped` 限制 YAW 运动。

6. OLED GIMBAL 页版本标记更新为：

```text
GYAW P11
```

### 当前 OLED 显示

```text
第 0 行：GYAW P11
第 2 行：C:<continuous x10> W:<wrapped x10>
第 4 行：T:<target continuous x10> N:<turn_count>
第 6 行：M:<mode> E:<enable> T5:<tick>
```

### 当前验收方式

```text
K2 短按：YAW 目标连续角 +5°
连续短按 K2：C 应能超过 900，即超过 90.0°
继续短按超过一圈时：W 应回绕到 0~3600 范围，N 应增加
K2 长按：StopHold，目标角应收敛到当前连续角
K3 短按/长按：Release，释放电机，但不清零 estimated_steps
```

### 当前边界

1. 当前仍是单轴 YAW，PITCH 电机未接入。
2. 当前只是连续位置命令和有效步数累计，还未实现位置 PD、RPM、加速度限制。
3. `GimbalStepperTest` 名字仍带 test，后续 P12/P13 应继续把它重命名或拆分为正式 `gimbal_stepper` 底层驱动。
4. 上电时采用“软件当前位置作为零点”策略，`position_valid = 1`。断电后物理角度不会保存。
