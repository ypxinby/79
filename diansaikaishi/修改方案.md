# 循迹小车比赛版任务封装与开发规划

## 一、总体目标

改进后的工程应满足以下目标：

1. **底层运控保持稳定**

   * 七路灰度循迹继续由 `CarController` 控制。
   * SEEK 航向找线继续使用陀螺仪目标 yaw。
   * 90° 转弯继续使用当前已经稳定的单轮转弯方案。
   * LOST_RECOVER、航向辅助、电机保护逻辑继续保留。

2. **上层任务可以自由组合**

   * Codex 创建新地图任务时，只需要填写动作序列。
   * 不需要修改电机、IMU、循迹 PD、90° 转弯等底层代码。
   * 一个任务可以由“找线、循迹、等待路口、左转、右转、停车”等动作自由组合。

3. **保留 OLED 实时调参**

   * 运行时可以继续调整循迹 KP、KD、速度、航向修正等安全参数。
   * 地图任务结构、动作顺序等参数在运行期间锁定，防止误操作。
   * OLED 能显示当前任务、当前动作、动作进度、传感器和故障信息。

4. **面向比赛可靠性**

   * 所有动作非阻塞执行。
   * 每个动作有超时保护。
   * 任务执行失败后能够安全停车，而不是一直转圈。
   * 不使用动态内存，不在单片机上解析复杂 JSON。
   * 新地图任务尽量只改一个任务配置文件。

当前工程已经明确由 `CarController` 统一输出电机速度，并具备 SEEK、FOLLOW_LINE、TURN_90、LOST_RECOVER 和航向控制，因此非常适合作为底层运动控制层继续封装。

---

# 二、推荐的整体分层架构

建议将工程分成五层。

```text
任务定义层 Mission Library
        ↓
任务管理层 Mission Manager
        ↓
动作执行层 Motion Action Executor
        ↓
底层运控层 CarController
        ↓
Motor / TrackSensor / IMU / HeadingControl
```

## 1. 底层驱动层

继续保留：

```text
motor.c
track_sensor.c
imu.c
heading_control.c
key.c
oled.c
encoder.c
```

职责保持不变：

* 读取硬件；
* 输出电机；
* 计算灰度误差；
* 更新 yaw；
* 生成航向修正量；
* 处理按键和 OLED 底层显示。

这一层不能知道“地图任务”是什么。

---

## 2. 底层运控层

继续由：

```text
car_controller.c
car_controller.h
```

负责：

* SEEK_LINE；
* FOLLOW_LINE；
* TURN_LEFT_90；
* TURN_RIGHT_90；
* LOST_RECOVER；
* 航向辅助；
* 电机输出；
* 运控超时与恢复。

但是需要进行一次重要的接口整理：

> `CarController` 只负责“如何执行一个运动命令”，不负责决定整张地图接下来应该做什么。

例如：

* 如何直线找线，由 `CarController` 负责；
* 找到线后是继续循迹还是左转，由动作层决定；
* 检测到右侧 90° 路口后是否右转，由任务层决定。

这样能够避免后续每增加一张地图，就往 `CarController` 中增加大量特殊状态。

---

## 3. 动作执行层

新增：

```text
motion_action.c
motion_action.h
motion_types.h
```

职责：

* 保存当前正在执行的动作；
* 启动对应的 `CarController` 运控模式；
* 检查动作完成条件；
* 检查动作超时；
* 返回成功、失败或超时；
* 动作切换时重置相关控制器状态。

动作层是整个封装的核心。

---

## 4. 任务管理层

新增：

```text
mission_manager.c
mission_manager.h
```

职责：

* 保存当前选择的任务；
* 保存当前执行到第几个动作；
* 启动、暂停、继续、取消任务；
* 接收动作执行结果；
* 成功后进入下一个动作；
* 失败后执行重试、跳过或安全停车；
* 任务全部完成后进入 `CAR_STATE_DONE`。

---

## 5. 任务定义层

新增：

```text
mission_library.c
mission_library.h
```

职责：

* 使用静态数组保存不同地图的动作组合；
* 保存任务名称；
* 保存任务动作数量；
* 保存任务所使用的参数配置；
* 提供任务注册表。

以后让 Codex 创建新任务时，原则上只允许修改：

```text
mission_library.c
```

正常情况下不再修改：

```text
motor.c
imu.c
heading_control.c
car_controller.c
```

# 三、动作模型设计

## 1. 第一版动作类型

建议第一版只加入已经验证稳定的动作。

```c
typedef enum {
    MOTION_ACTION_SEEK_LINE = 0,
    MOTION_ACTION_FOLLOW_LINE,
    MOTION_ACTION_TURN_LEFT_90,
    MOTION_ACTION_TURN_RIGHT_90,
    MOTION_ACTION_WAIT,
    MOTION_ACTION_STOP
} MotionActionType;
```

对应含义：

| 动作              | 作用                   |
| --------------- | -------------------- |
| `SEEK_LINE`     | 按目标 yaw 直线前进，直到检测到黑线 |
| `FOLLOW_LINE`   | 沿黑线循迹，直到满足指定结束条件     |
| `TURN_LEFT_90`  | 执行当前稳定的左 90° 单轮转弯    |
| `TURN_RIGHT_90` | 执行当前稳定的右 90° 单轮转弯    |
| `WAIT`          | 停车等待指定时间             |
| `STOP`          | 停车并结束任务              |

第一版不要立即加入过多动作，先确保任务执行框架可靠。

---

## 2. 动作执行状态

```c
typedef enum {
    MOTION_RESULT_IDLE = 0,
    MOTION_RESULT_RUNNING,
    MOTION_RESULT_SUCCESS,
    MOTION_RESULT_FAILED,
    MOTION_RESULT_TIMEOUT,
    MOTION_RESULT_CANCELLED
} MotionActionResult;
```

任何动作都必须返回明确结果。

不能只依赖：

```text
当前是不是还在运行
```

因为任务管理器必须知道动作究竟是：

* 正常完成；
* 找不到线；
* 转弯失败；
* 超时；
* 被用户取消。

---

## 3. FOLLOW_LINE 的结束条件

仅有一个普通的 `FOLLOW_LINE` 动作还不够，因为不同地图中“循迹到哪里结束”不同。

建议增加：

```c
typedef enum {
    FOLLOW_END_NEVER = 0,
    FOLLOW_END_LEFT_90_DETECTED,
    FOLLOW_END_RIGHT_90_DETECTED,
    FOLLOW_END_ANY_90_DETECTED,
    FOLLOW_END_START_LINE_DETECTED,
    FOLLOW_END_LINE_LOST,
    FOLLOW_END_DURATION,
    FOLLOW_END_LAP_COUNT
} FollowEndCondition;
```

例如：

```text
沿线走到左 90° 路口
沿线走到右 90° 路口
沿线走到起终点粗黑线
循迹 3000ms
循迹一圈
```

都可以使用同一个 `FOLLOW_LINE` 动作，只是结束条件不同。

---

## 4. 90° 检测和 90° 转弯必须分开

目前工程中，`FOLLOW_LINE` 检测到左、右 90° 后会直接进入转弯状态。这对固定地图很好，但不利于任务自由编排。

比赛版应逐步拆成：

```text
检测到左 90°路口
        ↓
向动作层上报 LEFT_90_DETECTED
        ↓
任务决定：
    左转
    继续直行
    停车
    将其作为计数标志
```

建议给循迹动作增加路口处理策略：

```c
typedef enum {
    TURN_POLICY_AUTO = 0,
    TURN_POLICY_REPORT_ONLY,
    TURN_POLICY_IGNORE
} TurnHandlingPolicy;
```

含义：

* `AUTO`：保持当前行为，检测到路口后自动转弯；
* `REPORT_ONLY`：只报告路口，不自动转；
* `IGNORE`：不把该路口作为动作结束条件。

为了保证重构期间不破坏现有稳定效果，第一阶段可以继续使用 `AUTO`；任务框架稳定后，再加入 `REPORT_ONLY`。

# 四、动作数据结构

推荐采用静态结构体和联合体，不使用动态内存。

```c
typedef enum {
    YAW_REFERENCE_CURRENT = 0,
    YAW_REFERENCE_MISSION_START,
    YAW_REFERENCE_ABSOLUTE
} YawReferenceType;

typedef struct {
    YawReferenceType reference;
    float angle_deg;
} MotionYawTarget;

typedef struct {
    MotionActionType type;

    uint32_t timeout_ms;
    uint8_t max_retries;

    union {
        struct {
            MotionYawTarget yaw;
            int16_t speed_override;
        } seek_line;

        struct {
            FollowEndCondition end_condition;
            TurnHandlingPolicy turn_policy;
            uint32_t duration_ms;
            uint8_t target_laps;
            int16_t speed_override;
        } follow_line;

        struct {
            int16_t speed_override;
            uint16_t min_turn_ms;
        } turn_90;

        struct {
            uint32_t wait_ms;
        } wait;
    } params;
} MotionAction;
```

## yaw 参考系必须明确

当前接口：

```c
CarController_SetSeekTargetYaw(float target_yaw_deg);
```

只接收一个浮点数，后续容易产生歧义：

* 是相对于本动作开始时的 yaw？
* 是相对于整场任务起点的 yaw？
* 还是 IMU 当前积分坐标中的绝对 yaw？

建议明确分成三种。

### `YAW_REFERENCE_CURRENT`

```text
目标 yaw = 动作开始时 yaw + angle_deg
```

适合：

```text
从当前位置向左偏 30° 找线
```

### `YAW_REFERENCE_MISSION_START`

```text
目标 yaw = 任务启动时 yaw + angle_deg
```

适合：

```text
整张地图统一使用任务起点作为 0°
```

这是比赛任务中最推荐的方式。

### `YAW_REFERENCE_ABSOLUTE`

```text
目标 yaw = angle_deg
```

只适合调试或已经明确管理 IMU yaw 坐标系的情况。

## 2. 使用宏提高可读性

在 `mission_library.h` 中定义：

```c
#define ACTION_SEEK_RELATIVE_YAW(yaw_deg, timeout) \
    { \
        .type = MOTION_ACTION_SEEK_LINE, \
        .timeout_ms = (timeout), \
        .max_retries = 1, \
        .params.seek_line = { \
            .yaw = { \
                .reference = YAW_REFERENCE_CURRENT, \
                .angle_deg = (yaw_deg) \
            }, \
            .speed_override = MOTION_USE_GLOBAL_SPEED \
        } \
    }
```

类似地定义：

```text
ACTION_FOLLOW_UNTIL_LEFT_90()
ACTION_FOLLOW_UNTIL_RIGHT_90()
ACTION_TURN_LEFT_90()
ACTION_TURN_RIGHT_90()
ACTION_WAIT_MS()
ACTION_STOP()
```

这样 Codex 编写新地图时，代码会接近自然语言，不容易误改底层。

# 六、任务管理器设计

## 1. 运行时结构

```c
typedef enum {
    MISSION_STATUS_IDLE = 0,
    MISSION_STATUS_READY,
    MISSION_STATUS_RUNNING,
    MISSION_STATUS_PAUSED,
    MISSION_STATUS_DONE,
    MISSION_STATUS_ERROR
} MissionStatus;

typedef struct {
    const MissionDefinition *definition;

    MissionStatus status;

    uint16_t current_action_index;
    uint8_t current_retry_count;

    uint32_t mission_elapsed_ms;
    uint32_t action_elapsed_ms;

    float mission_start_yaw;

    MotionActionResult last_action_result;
    uint16_t last_error_code;
} MissionRuntime;
```

---

## 2. 推荐接口

```c
void MissionManager_Init(void);

bool MissionManager_Select(uint8_t mission_id);
bool MissionManager_Start(void);

void MissionManager_Pause(void);
void MissionManager_Resume(void);
void MissionManager_Cancel(void);
void MissionManager_Reset(void);

void MissionManager_Update_20ms(void);

const MissionRuntime *MissionManager_GetRuntime(void);
```

---

## 3. 任务更新逻辑

```text
如果任务未运行：
    不执行动作

如果当前动作尚未启动：
    MotionAction_Start()

如果当前动作运行中：
    MotionAction_Update_20ms()

如果动作成功：
    current_action_index++
    启动下一个动作

如果动作超时或失败：
    如果 retry_count < max_retries：
        重启当前动作
    否则：
        停车
        进入 MISSION_STATUS_ERROR

如果执行到 STOP 或最后一个动作：
    停车
    进入 MISSION_STATUS_DONE
```

# 七、动作切换时必须统一重置的状态

每次动作开始或结束时，应集中清理：

```text
HeadingControl_Reset()
循迹 last_error 清零
航向锁定状态清零
turn_elapsed_ms 清零
recover_elapsed_ms 清零
丢线方向按需要重置
动作事件标志清零
动作完成标志清零
```

建议新增：

```c
void CarController_ResetTransientState(void);
```

不要让每个动作分别手动清理部分变量，否则后续容易出现：

```text
上一个动作的目标 yaw 残留
上一个动作的 PD 微分项影响新动作
转弯计时跨动作保留
LOST_RECOVER 方向跨任务保留
```

# 八、参数系统改进

## 1. 参数分成三个层级

### 第一层：全局控制参数

例如：

```text
track_kp
track_kd
base_speed
max_correction
heading_kp
heading_kd
heading_max_correction
turn_speed
turn_min_ms
```

用于控制车辆整体性能。

### 第二层：动作局部参数

例如：

```text
本次 SEEK 的目标 yaw
本次 FOLLOW 的结束条件
本次转弯的超时时间
本次动作是否覆盖默认速度
```

仅对当前动作生效。

### 第三层：任务参数

例如：

```text
任务名称
动作顺序
动作数量
任务目标圈数
任务使用的控制配置
```

任务参数不能在任务运行中随意修改。

---

## 2. 全局参数与动作覆盖规则

推荐采用：

```text
动作没有指定覆盖值：
    使用 AppConfig 中的全局参数

动作指定了覆盖值：
    仅当前动作使用覆盖值
```

例如：

```c
#define MOTION_USE_GLOBAL_SPEED (-1)
```

```text
speed_override = -1：
    使用全局 base_speed

speed_override = 120：
    当前动作固定使用 120
```

这样可以：

* 继续通过 OLED 调整全局速度；
* 对某个危险动作单独降低速度；
* 不需要为每个地图复制一套 KP、KD。

# 九、OLED 实时调参设计

## 1. 不建议继续扩大 menu.c 中的 switch-case

当前参数较少时，直接判断参数名称还可以；比赛版参数增加后容易变得混乱。

建议新增：

```text
parameter_registry.c
parameter_registry.h
```

参数描述结构：

```c
typedef enum {
    PARAM_APPLY_LIVE = 0,
    PARAM_APPLY_NEXT_ACTION,
    PARAM_APPLY_LOCKED_WHILE_RUNNING
} ParamApplyPolicy;

typedef struct {
    const char *name;
    void *value_ptr;

    ParamValueType value_type;

    float min_value;
    float max_value;
    float step;

    ParamApplyPolicy apply_policy;
    ParamGroup group;
} ParamDescriptor;
```

---

## 2. 参数应用策略

### `PARAM_APPLY_LIVE`

运行中立即生效：

```text
track_kp
track_kd
base_speed
max_correction
heading_kp
heading_kd
heading_max_correction
```

### `PARAM_APPLY_NEXT_ACTION`

修改后从下一个动作开始生效：

```text
seek_heading_offset
turn_speed
turn_min_ms
recover_speed
```

### `PARAM_APPLY_LOCKED_WHILE_RUNNING`

任务运行时禁止修改：

```text
selected_mission
动作顺序
动作超时
任务圈数
失败策略
```

OLED 上可以显示：

```text
LIVE
NEXT
LOCK
```

---

## 3. 任务目标 yaw 与实时 yaw 微调必须分开

不要继续只使用一个 `YAW` 参数同时表达：

```text
任务本身要求的目标角度
现场调车产生的航向微调
```

建议拆成：

```text
action_target_yaw
heading_target_trim_deg
```

最终目标：

```text
effective_target_yaw
= action_target_yaw
+ heading_target_trim_deg
```

其中：

* `action_target_yaw` 来自任务定义；
* `heading_target_trim_deg` 可以通过 OLED 实时微调；
* 切换到下一个动作时，可以选择保留或清零 trim；
* 不会因为现场调参而破坏任务原始角度定义。

这是比赛版非常重要的一项设计。

# 十、按键方案

建议尽量保留当前按键习惯。

## 正常状态

```text
K1 短按：切换 OLED 页面
K1 长按：进入参数菜单

K2 短按：
    READY -> 启动选择的任务
    RUNNING -> 暂停
    PAUSED -> 继续

K3 短按：
    进入 READY 或返回

K3 长按：
    紧急停车
    清除任务运行状态
    返回 READY
```

## 参数菜单

```text
K1 短按：下一个参数
K1 长按：退出菜单
K2：增加
K3：减少
```

将任务选择作为参数菜单中的第一项：

```text
TASK
PROFILE
LAP
SPD
KP
KD
...
```

这样无需破坏当前 K2 启动逻辑。

# 十一、暂停与恢复策略

暂停时必须：

```text
电机立即停止
动作计时暂停
任务计时暂停
航向控制暂时关闭
```

继续运行时建议默认：

```text
重新初始化当前动作
重新锁定航向目标
重置循迹微分项
重新确认当前传感器状态
```

不建议直接保留暂停前的控制器瞬时状态，因为暂停期间可能：

* 手动移动了小车；
* 改变了车身角度；
* 调整了参数；
* 灰度传感器所在位置发生变化。

建议增加动作恢复策略：

```c
typedef enum {
    ACTION_RESUME_CONTINUE = 0,
    ACTION_RESUME_RESTART
} ActionResumePolicy;
```

第一版全部使用：

```text
ACTION_RESUME_RESTART
```

更加安全。

---

# 十二、故障和超时机制

每个动作必须配置超时。

示例：

```text
SEEK_LINE：3000ms
FOLLOW_LINE：根据地图设置 5000～15000ms
TURN_90：1400～1800ms
WAIT：由等待时间决定
```

建议错误码：

```c
typedef enum {
    MOTION_ERROR_NONE = 0,
    MOTION_ERROR_SEEK_TIMEOUT,
    MOTION_ERROR_FOLLOW_TIMEOUT,
    MOTION_ERROR_TURN_TIMEOUT,
    MOTION_ERROR_LINE_LOST,
    MOTION_ERROR_IMU_NOT_READY,
    MOTION_ERROR_INVALID_ACTION,
    MOTION_ERROR_INVALID_MISSION
} MotionErrorCode;
```

默认失败策略：

```text
动作允许重试一次
仍失败则立即停车
进入 ERROR
OLED 显示动作编号和错误原因
```

不要在未知状态下自动无限重试。

---

# 十三、任务合法性检查

建议在启动任务前运行：

```c
bool Mission_Validate(const MissionDefinition *mission);
```

检查：

1. 任务指针不能为空；
2. 动作数量不能为 0；
3. 动作数量不能超过最大值；
4. 所有动作类型必须合法；
5. 必要动作必须设置超时；
6. SEEK 的 yaw 范围必须合理；
7. FOLLOW 的结束条件必须合法；
8. 最后一个动作必须是 STOP；
9. 重试次数不能超过上限；
10. 参数覆盖值必须在安全范围内。

这样 Codex 即使生成了错误任务，也能在运行前被发现。

# 十五、20ms 调度顺序调整

建议调整为：

```text
1. Imu_Update(0.02f)

2. Key_Update_20ms()
3. Key_GetEvent()
4. Menu_HandleKeyEvent()

5. MissionManager_Update_20ms()
   └─ MotionAction_Update_20ms()
      └─ 向 CarController 设置当前运动命令

6. CarController_Update_20ms()
   └─ 读取灰度传感器
   └─ 更新循迹/航向/转弯控制
   └─ 输出左右电机
   └─ 更新动作反馈和传感器事件

7. 更新调试变量

8. OledUi_Update_20ms()
```

任务管理器和动作执行器不能直接调用：

```text
Motor_SetLeftSpeed()
Motor_SetRightSpeed()
```

电机输出必须继续只有一个所有者：

```text
CarController
```

# 十七、比赛版参数 Profile

后续可以增加：

```c
typedef struct {
    const char *name;

    int16_t base_speed;
    int16_t search_speed;
    int16_t turn_speed;

    int16_t track_kp;
    int16_t track_kd;

    int16_t heading_kp;
    int16_t heading_kd;
    int16_t heading_max_correction;
} ControlProfile;
```

例如：

```text
STABLE：稳定低速
FAST：比赛高速
LOW_BATTERY：低电压补偿
SMOOTH_FLOOR：光滑地面
ROUGH_FLOOR：粗糙地面
```

任务只引用 Profile，不复制控制参数。

```text
MAP-A + STABLE
MAP-A + FAST
MAP-B + STABLE
```

这样地图任务和车辆调参相互独立。

# 十八、不推荐的实现方式

## 1. 不要为每张地图写一个新的大状态机

错误方式：

```text
MAP_A_STATE_1
MAP_A_STATE_2
MAP_A_STATE_3
MAP_B_STATE_1
MAP_B_STATE_2
...
```

地图增加后会产生状态爆炸。

---

## 2. 不要让任务代码直接控制电机

错误方式：

```c
Motor_SetLeftSpeed(0);
Motor_SetRightSpeed(130);
```

任务只应该提出：

```text
执行右 90° 转弯
```

具体电机输出仍由 `CarController` 决定。

---

## 3. 不要使用阻塞延时

错误方式：

```c
turn_right();
delay_ms(800);
follow_line();
```

这样会导致：

* OLED 不刷新；
* 按键无法响应；
* IMU yaw 停止更新；
* 紧急停车失效；
* 控制周期被破坏。

所有动作必须以 20ms 周期非阻塞执行。

---

## 4. 不要在 MCU 上解析复杂 JSON

比赛版推荐：

```text
PC 上由 Codex 生成静态 C 动作数组
MCU 直接编译执行
```

而不是：

```text
MCU 上读取 JSON
动态创建动作
动态分配内存
```

静态 C 数组更容易检查、调试和保证可靠性。

---

## 5. 不要把任务目标 yaw 和现场调车 yaw 混为一个参数

必须区分：

```text
任务角度
现场微调
当前动作起点
任务起点
IMU 绝对积分值
```

否则地图动作增加后，很容易出现方向基准混乱。

---

# 十九、推荐的最终使用流程

完成封装后，开发一张新地图的流程应当是：

```text
1. 分析地图关键节点。

2. 将地图拆成动作：
   找线
   循迹到右路口
   右转
   循迹到左路口
   左转
   循迹到终点
   停车

3. 在 mission_library.c 中增加动作数组。

4. 在任务注册表中增加任务名称。

5. 编译下载。

6. OLED 选择任务。

7. 运行并观察当前动作。

8. 通过 OLED 调整 KP、KD、速度和 yaw trim。

9. 固化稳定参数或保存为 Profile。
```

Codex 创建新任务时，只需要获得：

```text
地图节点顺序
每个路口方向
每段结束条件
每个动作超时时间
目标 yaw
是否使用特殊速度
失败后的处理方式
```

# 二十一、方案结论

当前稳定版本应当被视为：

```text
运动执行底座
```

而不是继续扩展成：

```text
包含所有地图逻辑的超级状态机
```

最合理的比赛版结构是：

```text
Mission Library：描述要做什么
Mission Manager：决定下一步做什么
Motion Action：判断当前动作是否完成
CarController：决定动作具体怎么执行
Motor/IMU/Sensor：完成硬件输入输出
```

这样以后 Codex 创建一个新任务时，主要工作将从：

```text
修改底层状态机和电机逻辑
```

变成：

```text
添加一个可读、可检查的动作数组
```

既能保留现有循迹、90° 转弯和陀螺仪辅助运控的稳定效果，也能满足不同比赛地图快速更换任务的需求。
