# K230视觉-二维云台系统联调与参数调试手册 V1.0

面向：视觉算法开发人员  
适用阶段：K230视觉端开发、PC发送器联调、MSPM0二维云台视觉追踪参数调试  
协议版本：Vision Target Protocol V1

本文档只描述视觉端需要遵守的输出接口、通信协议、联调流程和参数调试方法。视觉端不需要了解MSPM0底层电机控制细节，也不应向云台发送角度、速度、PID或STEP类控制命令。

## 1. 系统架构

完整链路如下：

```text
K230
↓
Vision Protocol V1
↓
MSPM0 UART
↓
VisionReceiver
↓
GimbalVisionAdapter
↓
YAW/PITCH Vision Tracker
↓
双轴云台
```

职责划分：

| 模块 | 职责 |
| --- | --- |
| K230视觉端 | 检测目标、输出目标中心、控制发送频率、保证视觉结果稳定 |
| Vision Protocol V1 | 把视觉目标观测编码成固定40字节二进制帧 |
| MSPM0 VisionReceiver | UART接收、帧同步、长度检查、CRC校验、sequence/session管理 |
| GimbalVisionAdapter | 将图像坐标转换为 `error_x/error_y` |
| YAW/PITCH Vision Tracker | 根据误差和调参参数生成云台运动命令 |
| 双轴云台 | 执行运动、限位、保护、停止保持 |

视觉端只输出“目标观测”。云台端负责把误差转换成运动，并负责限位和保护。

## 2. K230输出接口

视觉端每一帧必须输出以下信息：

| 字段 | 说明 |
| --- | --- |
| `frame_width` | 当前图像宽度，单位px |
| `frame_height` | 当前图像高度，单位px |
| `target_center_x` | 目标中心X坐标 |
| `target_center_y` | 目标中心Y坐标 |
| `valid` | 当前是否有有效目标 |

可选输出：

| 字段 | 说明 |
| --- | --- |
| `confidence` | 置信度，推荐范围 `0~1000` |
| `bbox` | 目标框 `(x, y, width, height)` |
| `target_id` | 多目标跟踪ID |

禁止视觉端输出或控制：

- 云台角度；
- 电机速度；
- PID计算结果；
- STEP数量；
- STEP/DIR/EN；
- 云台启停控制命令。

视觉端只负责告诉云台：“目标在图像里的哪里、当前有没有目标、结果可信度如何”。

## 3. 视觉协议说明

使用 `Vision Target Protocol V1`，固定40字节二进制帧，不使用JSON、字符串或浮点数。

基本要求：

```text
UART: 115200 baud, 8N1, no flow control
Frame length: 40 bytes
Byte order: little-endian
Protocol version: 1
Message type: 1 = TARGET_REPORT
Payload length: 30
CRC: CRC-16/CCITT-FALSE
```

### 3.1 字段表

| Offset | 长度 | 类型 | 字段 | 说明 |
| ---: | ---: | --- | --- | --- |
| 0 | 1 | `uint8` | `magic_0` | 固定 `0xA5` |
| 1 | 1 | `uint8` | `magic_1` | 固定 `0x5A` |
| 2 | 1 | `uint8` | `version` | 固定 `0x01` |
| 3 | 1 | `uint8` | `message_type` | 固定 `0x01` |
| 4 | 1 | `uint8` | `flags` | 目标有效、bbox、ID、confidence等标志 |
| 5 | 1 | `uint8` | `reserved` | V1固定 `0` |
| 6 | 2 | `uint16` | `payload_length` | V1固定 `30` |
| 8 | 4 | `uint32` | `session_id` | 视觉程序本次运行会话ID，非零 |
| 12 | 2 | `uint16` | `sequence` | 每发送一帧递增，允许 `65535→0` |
| 14 | 4 | `uint32` | `timestamp_ms` | K230程序启动后的毫秒时间 |
| 18 | 2 | `uint16` | `frame_width` | 图像宽度 |
| 20 | 2 | `uint16` | `frame_height` | 图像高度 |
| 22 | 2 | `uint16` | `target_center_x` | 目标中心X |
| 24 | 2 | `uint16` | `target_center_y` | 目标中心Y |
| 26 | 2 | `uint16` | `confidence` | 置信度，正常范围 `0~1000` |
| 28 | 2 | `uint16` | `target_id` | 目标ID，无ID时 `0xFFFF` |
| 30 | 2 | `uint16` | `bbox_x` | bbox左上角X |
| 32 | 2 | `uint16` | `bbox_y` | bbox左上角Y |
| 34 | 2 | `uint16` | `bbox_width` | bbox宽度 |
| 36 | 2 | `uint16` | `bbox_height` | bbox高度 |
| 38 | 2 | `uint16` | `crc16` | CRC，小端存放 |

`payload_length=30` 表示 offset 8 的 `session_id` 到 offset 37 的 `bbox_height` 共30字节，不包含帧头、版本、类型、flags、reserved、length字段，也不包含CRC本身。

### 3.2 flags定义

| bit | 名称 | 含义 |
| ---: | --- | --- |
| 0 | `TARGET_VALID` | 当前有有效目标 |
| 1 | `HAS_BBOX` | bbox字段有效 |
| 2 | `HAS_TARGET_ID` | target_id字段有效 |
| 3 | `HAS_CONFIDENCE` | confidence字段有效 |
| 4 | `SOURCE_RESTART` | 视觉端刚启动或新会话开始 |
| 5~7 | reserved | V1必须为0 |

常用组合：

```text
0x00: 无目标
0x01: 有目标中心
0x09: 有目标中心 + confidence
0x0F: 有目标中心 + bbox + target_id + confidence
0x1F: 新会话启动阶段，且目标字段均有效
```

### 3.3 session_id规则

`session_id` 用于识别K230视觉程序是否重启。

要求：

- 每次K230程序启动生成新的非零 `uint32 session_id`；
- 同一次程序运行期间保持不变；
- 不允许简单使用 `timestamp_ms` 作为 `session_id`；
- 推荐使用随机数或随机数加启动计数；
- MSPM0检测到 `session_id` 变化后，会重新初始化sequence状态。

### 3.4 sequence规则

要求：

- 每发送一帧递增一次；
- `valid=0` 的无目标帧也必须递增；
- `65535` 后自然回绕到 `0`；
- 不要在同一 `session_id` 下随意把sequence清零。

MSPM0使用有符号差值判断新旧帧，因此可以正确处理回绕。

### 3.5 CRC说明

CRC算法：

```text
CRC-16/CCITT-FALSE
poly   = 0x1021
init   = 0xFFFF
refin  = false
refout = false
xorout = 0x0000
```

计算范围：

```text
offset 0 ~ offset 37，共38字节
不包含offset 38~39的CRC自身
```

CRC结果以小端写入帧尾。

Python参考实现：

```python
def crc16_ccitt_false(data: bytes) -> int:
    crc = 0xFFFF
    for byte in data:
        crc ^= byte << 8
        for _ in range(8):
            if crc & 0x8000:
                crc = ((crc << 1) ^ 0x1021) & 0xFFFF
            else:
                crc = (crc << 1) & 0xFFFF
    return crc
```

## 4. 坐标与误差定义

图像坐标系：

```text
左上角为 (0, 0)
X 向右增加
Y 向下增加
单位：px
```

MSPM0适配层计算：

```text
error_x = target_center_x - frame_width / 2
error_y = target_center_y - frame_height / 2
```

示例：

```text
图像尺寸：640 × 480
目标中心：(400, 200)

error_x = 400 - 320 = 80
error_y = 200 - 240 = -40
```

误差对应关系：

| 误差 | 控制轴 |
| --- | --- |
| `error_x` | YAW |
| `error_y` | PITCH |

当前实机方向已经校验：

```text
error_x > 0 → YAW正方向运动
error_x < 0 → YAW反方向运动
error_y > 0 → PITCH一个方向运动
error_y < 0 → PITCH反方向运动
```

视觉端不要自行输出角度补偿，只要保证坐标定义正确。

## 5. 视觉质量要求

### 5.1 FPS要求

推荐：

```text
20~30 FPS
```

最低：

```text
10 FPS
```

低于10FPS时，云台容易出现一段一段地追踪，甚至触发通信超时保护。

### 5.2 坐标稳定性

检测框中心抖动会直接变成 `error_x/error_y` 抖动，最终表现为云台抖动。

视觉侧建议：

- 对检测框中心做适度平滑；
- 多目标时保持目标选择稳定，不要频繁切换目标；
- 检测短暂丢失时可谨慎保持目标，但不要长期伪造目标；
- bbox中心应使用稳定定义，例如 `(x + w/2, y + h/2)`；
- 置信度过低时建议发送 `valid=0`，而不是发送乱跳坐标。

### 5.3 延迟要求

视觉延迟会直接影响追踪效果。延迟越大，越容易出现滞后、过冲和来回摆动。

建议：

- 控制推理耗时；
- 保持固定发送周期；
- 不要积压旧帧；
- K230端应优先发送最新检测结果；
- 如果算法输出已经明显过期，建议发送无目标帧或丢弃旧结果。

## 6. 云台参数调节说明

当前支持通过PC Python工具在线调参。调参命令通过同一视觉UART发送，不需要修改MSPM0代码或重新烧录。

注意：

- 参数暂不保存到Flash；
- 断电或复位后恢复默认值；
- 调参只影响视觉追踪参数，不改变视觉协议；
- YAW和PITCH参数相互独立。

### 6.1 PITCH参数

工具：

```text
diansaikaishi/tools/vision_pitch_tuning_cli.py
```

命令示例：

```powershell
python diansaikaishi\tools\vision_pitch_tuning_cli.py --port COM7 GET PARAM
python diansaikaishi\tools\vision_pitch_tuning_cli.py --port COM7 SET DB 8
python diansaikaishi\tools\vision_pitch_tuning_cli.py --port COM7 SET KP 0.060
python diansaikaishi\tools\vision_pitch_tuning_cli.py --port COM7 SET MAXSPD 6.0
python diansaikaishi\tools\vision_pitch_tuning_cli.py --port COM7 SET TIMEOUT 300
python diansaikaishi\tools\vision_pitch_tuning_cli.py --port COM7 DEFAULT
```

命令格式：

```text
$VPT GET PARAM
$VPT SET DB <px>
$VPT SET KP <value>
$VPT SET MAXSPD <deg/s>
$VPT SET TIMEOUT <ms>
$VPT DEFAULT
```

参数含义：

| 参数 | 含义 | 现象 |
| --- | --- | --- |
| `DB` | deadband，中心误差范围内不运动 | 太小易抖，太大居中误差变大 |
| `KP` | 误差转速度比例 | 太小跟随慢，太大易抖或过冲 |
| `MAXSPD` | 最大追踪速度 | 太小跟不上，太大运动冲 |
| `TIMEOUT` | 视觉数据超时保护 | 太小易误停，太大丢帧后停得慢 |

### 6.2 YAW参数

工具：


命令示例：



YAW参数含义与PITCH一致，只是作用在 `error_x → YAW` 链路。

## 7. 参数调节方法

调参原则：

```text
一次只修改一类参数
每次修改后记录现象
先调稳定性，再调速度
先单轴，再双轴
```

### 第一步：固定目标，调deadband

测试方法：

- 固定目标在画面中心附近；
- 保持K230连续发送；
- 观察云台是否小幅来回抖动。

调整：

- 如果目标不动但云台抖动，增大 `DB`；
- 如果目标明显偏离中心但云台不动，减小 `DB`。

建议：

```text
从 DB=8 开始
每次调整 2px
```

### 第二步：移动目标，调Kp

测试方法：

- 缓慢水平移动目标，观察YAW；
- 缓慢垂直移动目标，观察PITCH；
- 看云台是否明显跟不上。

调整：

- 跟随太慢：增大 `KP`；
- 来回摆动、过冲明显：减小 `KP`；
- 坐标本身抖动时，先处理视觉稳定性，不要只靠减小Kp硬压。

建议：

```text
YAW 每次调整 0.010
PITCH 每次调整 0.010
```

### 第三步：快速移动目标，调MAXSPD

测试方法：

- 让目标快速跨越画面中心；
- 观察是否跟不上；
- 观察是否运动太冲或机械声音异常。

调整：

- 跟不上：适当增大 `MAXSPD`；
- 运动冲、容易过冲或声音异常：减小 `MAXSPD`。

注意：`MAXSPD` 不能无限增大，应以实机稳定、不失步、不发热异常为准。

### 第四步：通信测试，调TIMEOUT

测试方法：

- 正常发送目标帧；
- 停止发送；
- 观察云台是否在合理时间内停止。

调整：

- 偶发丢帧导致误停：适当增大 `TIMEOUT`；
- 断流后停得太慢：减小 `TIMEOUT`。

建议：

```text
从 300ms 开始
一般不要低于 100ms
```

## 8. 推荐初始参数

当前实机通过参数：

| 轴 | DB | KP | MAXSPD | TIMEOUT |
| --- | ---: | ---: | ---: | ---: |
| YAW | 8 px | 0.080 | 24 deg/s | 300 ms |
| PITCH | 8 px | 0.060 | 6 deg/s | 300 ms |

如果视觉坐标较抖，优先增大 `DB` 或在视觉侧做坐标平滑；不要一开始就大幅降低 `KP`。

## 9. 联调测试流程

联调前确认：

- MSPM0已烧录支持Vision Protocol V1和双轴视觉追踪的程序；
- YAW和PITCH已完成零点确认；
- YAW world lock关闭；
- K230或PC发送器使用115200 8N1；
- `session_id` 非零；
- `sequence` 连续递增。

### 测试1：中心目标

输入：

```text
640×480
target_center = (320, 240)
valid = 1
```

预期：

- `error_x≈0`
- `error_y≈0`
- 云台基本不动；
- 若抖动明显，优先调 `DB` 或检查视觉坐标抖动。

### 测试2：水平移动

输入：

```text
只改变 x
y 保持 240
```

预期：

- YAW运动；
- PITCH基本保持；
- `error_x>0` 和 `error_x<0` 应对应两个相反方向。

PC发送器示例：

```powershell
python diansaikaishi\tools\vision_protocol_sender.py --mode valid --port COM7 --fps 20 --count 100 --session-id 0x20000001 --sequence 0 --x 420 --y 240
python diansaikaishi\tools\vision_protocol_sender.py --mode valid --port COM7 --fps 20 --count 100 --session-id 0x20000002 --sequence 0 --x 220 --y 240
```

### 测试3：垂直移动

输入：

```text
只改变 y
x 保持 320
```

预期：

- PITCH运动；
- YAW基本保持；
- `error_y>0` 和 `error_y<0` 应对应两个相反方向。

PC发送器示例：

```powershell
python diansaikaishi\tools\vision_protocol_sender.py --mode valid --port COM7 --fps 20 --count 100 --session-id 0x20000003 --sequence 0 --x 320 --y 340
python diansaikaishi\tools\vision_protocol_sender.py --mode valid --port COM7 --fps 20 --count 100 --session-id 0x20000004 --sequence 0 --x 320 --y 140
```

### 测试4：斜向移动

输入：

```text
x/y 同时变化
```

预期：

- YAW和PITCH同时响应；
- 两轴状态都进入追踪；
- 没有互相覆盖或一轴抢占另一轴。

PC发送器示例：

```powershell
python diansaikaishi\tools\vision_protocol_sender.py --mode valid --port COM7 --fps 20 --count 100 --session-id 0x20000005 --sequence 0 --x 420 --y 340
```

### 测试5：目标丢失

输入：

```text
valid = 0
```

预期：

- 云台停止；
- 不继续使用最后一帧非零速度；
- 通信仍正常，不应显示为完全断连。

PC发送器示例：

```powershell
python diansaikaishi\tools\vision_protocol_sender.py --mode no-target --port COM7 --fps 20 --count 20 --session-id 0x20000006 --sequence 0
```

## 10. 常见问题排查

### 10.1 云台只动一下

重点检查：

- K230是否只发送了一帧；
- FPS是否过低；
- `sequence` 是否连续递增；
- `session_id` 是否非零且同一运行期间保持不变；
- 是否超过 `TIMEOUT` 后进入超时停止；
- CRC是否正确。

PC发送器常见坑：

```text
--count 默认可能只发少量帧
持续测试时请显式设置 --count 和 --fps
```

示例：

```powershell
python diansaikaishi\tools\vision_protocol_sender.py --mode valid --port COM7 --fps 20 --count 100 --x 420 --y 240
```

### 10.2 云台抖动

重点检查：

- 检测框中心是否抖动；
- `DB` 是否太小；
- `KP` 是否太大；
- 视觉端是否频繁切换目标；
- 目标中心计算是否稳定；
- K230输出是否有延迟或帧间隔不均匀。

处理顺序：

```text
先看坐标稳定性
再调 DB
再调 KP
最后才调 MAXSPD
```

### 10.3 方向反

重点检查：

- 图像坐标是否按左上角原点；
- X是否向右增加；
- Y是否向下增加；
- `target_center_x/y` 是否填反；
- 图像是否被镜像或旋转；
- K230显示坐标和发送坐标是否一致。

视觉端不要通过发送云台角度修正方向；方向问题应先从坐标定义和图像预处理查起。

### 10.4 延迟大

重点检查：

- K230推理时间；
- 发送频率；
- 是否积压旧帧；
- 串口是否阻塞；
- 是否每次发送最新结果；
- bbox平滑是否过重。

建议：

- 保持20~30FPS；
- 不要发送过期检测结果；
- 推理和发送之间尽量减少排队；
- 如需平滑，使用轻量平滑，不要引入明显滞后。

### 10.5 无目标后云台没有停

重点检查：

- 是否真的发送 `flags bit0 = 0`；
- 无目标帧是否仍然CRC正确；
- 无目标帧的 `sequence` 是否继续递增；
- 是否误把低置信度目标仍作为 `valid=1` 发送；
- 是否K230断流但没有发送无目标帧。

推荐策略：

```text
检测不到目标，但视觉程序还活着：发送 valid=0 无目标帧
视觉程序停止或串口断开：MSPM0会按 TIMEOUT 进入保护停止
```

## 11. K230端实现建议

K230端推荐主循环：

```python
session_id = generate_nonzero_random_u32()
sequence = 0
start_time = monotonic_ms()

while True:
    image = camera.read()
    result = detect_target(image)
    timestamp_ms = monotonic_ms() - start_time

    if result.valid:
        frame = build_valid_frame(
            session_id=session_id,
            sequence=sequence,
            timestamp_ms=timestamp_ms,
            frame_width=image.width,
            frame_height=image.height,
            target_center_x=result.cx,
            target_center_y=result.cy,
            confidence=result.confidence,
            bbox=result.bbox,
        )
    else:
        frame = build_no_target_frame(
            session_id=session_id,
            sequence=sequence,
            timestamp_ms=timestamp_ms,
            frame_width=image.width,
            frame_height=image.height,
        )

    uart.write(frame)
    sequence = (sequence + 1) & 0xFFFF
    wait_until_next_frame()
```

实现重点：

- 不要用字符串发送；
- 不要发送浮点数；
- 不要漏算CRC；
- 不要重复使用旧sequence；
- 不要在同一session下随意清零sequence；
- K230程序重启后必须换新的 `session_id`。

## 12. 最小验收标准

视觉端完成后，应至少通过以下验收：

| 项目 | 预期 |
| --- | --- |
| 固定中心目标 | 云台不明显抖动 |
| 水平移动 | 只有YAW明显响应 |
| 垂直移动 | 只有PITCH明显响应 |
| 斜向移动 | YAW/PITCH同时响应 |
| `valid=0` | 云台停止 |
| 停止发送 | 超时后云台停止 |
| K230重启 | 新 `session_id` 后恢复接收 |
| sequence回绕 | `65535→0` 正常接收 |
| 错误CRC | MSPM0丢弃，不更新目标 |

以上通过后，再进入更长时间的真实目标追踪、遮挡恢复和参数细调。
