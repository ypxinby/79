# HC-SR04 超声波与舵机扫描避障扩展方案

## 1. 改造目标

在现有电赛小车工程中增加：

```text
HC-SR04 超声波测距
+
舵机左右扫描
+
前方障碍物检测
+
停车等待
+
左右空间判断
+
后续可扩展主动绕障
```

现有工程架构为：

```text
MissionLibrary
      ↓
MissionManager
      ↓
MotionAction
      ↓
CarController
      ↓
Motor
```

改造后建议增加：

```text
Ultrasonic       HC-SR04 底层测距
Servo            舵机角度控制
ObstacleScanner  舵机扫描和距离融合
ObstacleMonitor  障碍物状态判断
ObstacleAvoidance 避障动作状态机
```

完整关系：

```text
HC-SR04
   ↓
Ultrasonic
   ↓
ObstacleScanner ← Servo
   ↓
ObstacleMonitor
   ↓
MotionAction / ObstacleAvoidance
   ↓
CarController
   ↓
Motor
```

必须继续遵守现有原则：

```text
超声波模块不直接调用 Motor_SetSpeed()
舵机扫描模块不直接控制左右电机
ObstacleAvoidance 只能通过 CarController 请求运动
最终电机输出仍统一由 CarController 完成
```

当前工程已经采用 `CarController—MotionAction—MissionManager—MissionLibrary` 分层，并规定任务层不得直接调用电机接口，因此新增避障功能也应继续沿用该边界。

---

# 2. 功能分级

避障功能不要一次性全部完成，建议分成三个等级。

## 2.1 一级：前方障碍物停车

小车正常循迹或 SEEK 时，超声波固定朝前。

检测到障碍物：

```text
减速
→
停车
→
等待障碍物消失
→
继续原任务
```

这是最先完成的功能，也是比赛安全价值最高的功能。

建议首先验证：

```text
前方 20～30 cm 有障碍物时能够稳定停车
障碍物移开后能够继续原来的任务
停车期间任务计时冻结
停车期间不会误进入下一个动作
```

---

## 2.2 二级：停车后舵机左右扫描

前方发现障碍物后：

```text
停车
→
舵机看左侧
→
测量左侧距离
→
舵机看右侧
→
测量右侧距离
→
回到正前方
→
判断哪侧更空旷
```

这一阶段只做环境判断，不立即实施复杂绕行。

可在 OLED 上显示：

```text
F:前方距离
L:左侧距离
R:右侧距离
DIR:推荐方向
```

例如：

```text
F:18 L:85 R:42
DIR:L
```

表示前方 18 cm，左侧 85 cm，右侧 42 cm，推荐从左侧绕行。

---

## 2.3 三级：主动绕过障碍物并重新找线

完整主动绕障需要小车具备：

```text
指定角度转向
保持航向直行
按时间或距离前进
绕过障碍物
重新寻找黑线
恢复原任务
```

当前工程已有：

```text
IMU yaw
HeadingControl
SEEK 航向保持
灰度找线
任务动作系统
```

但还缺少稳定的：

```text
IMU 指定角度转向
按距离前进
绕障后的线路重捕获
```

因此建议把主动绕障放在前方停车和舵机扫描稳定之后开发。

---

# 3. 硬件连接方案

## 3.1 HC-SR04 接线

HC-SR04 常见接口：

```text
VCC
TRIG
ECHO
GND
```

建议接法：

```text
VCC  → 5V
GND  → 小车公共 GND
TRIG → MSPM0 普通 GPIO 输出
ECHO → MSPM0 定时器捕获引脚或 GPIO 中断引脚
```

## 3.2 ECHO 电平问题

HC-SR04 的 ECHO 通常输出接近 5V，MSPM0G3507 GPIO 为 3.3V 系统，**不要直接把 ECHO 接到主控引脚**。

建议使用电阻分压：

```text
HC-SR04 ECHO
      |
     1kΩ
      |
      +------ MCU ECHO 输入
      |
     2kΩ
      |
     GND
```

分压后的电压约为：

```text
5V × 2 / (1 + 2) ≈ 3.33V
```

也可以使用电平转换模块。

TRIG 是否需要电平转换应结合实际模块验证，通常 3.3V GPIO 高电平能够被识别，但工程上仍应通过实测确认。

---

## 3.3 舵机供电

舵机不能直接使用 MSPM0 的 3.3V 供电，也不建议直接由开发板的小电流稳压输出供电。

建议：

```text
舵机 VCC    → 独立 5V～6V 电源
舵机 GND    → 小车公共 GND
舵机 Signal → MCU PWM 引脚
```

注意：

```text
舵机电源与主控必须共地
舵机电源要有足够电流
舵机启动和堵转电流明显高于静态电流
```

建议在舵机电源附近增加：

```text
470 μF～1000 μF 电解电容
+
0.1 μF 陶瓷电容
```

避免舵机运动造成：

```text
主控复位
OLED 闪烁
MPU6050 数据异常
电机输出抖动
```

---

## 3.4 引脚和定时器分配原则

具体引脚需要根据当前 `empty.syscfg` 中的资源占用确定。

建议分配：

```text
TRIG：任意空闲 GPIO 输出
ECHO：支持定时器捕获或外部中断的 GPIO
SERVO：支持硬件 PWM 的定时器通道
```

不要与以下资源冲突：

```text
电机 PWM
100 μs 软件 PWM Tick
MPU6050 模拟 I2C
OLED 接口
三个按键
七路灰度传感器
```

舵机优先使用硬件 PWM，不要在 20 ms 主循环里通过阻塞延时产生舵机脉冲。

---

# 4. 新增文件结构

建议新增：

```text
ultrasonic.c / ultrasonic.h
servo.c / servo.h
obstacle_scanner.c / obstacle_scanner.h
obstacle_monitor.c / obstacle_monitor.h
obstacle_avoidance.c / obstacle_avoidance.h
```

职责如下。

## 4.1 ultrasonic

负责：

```text
产生 TRIG 脉冲
检测 ECHO 上升沿
检测 ECHO 下降沿
计算高电平时间
换算距离
处理测量超时
标记数据有效性
```

不负责：

```text
控制舵机
判断向左还是向右
控制电机
决定是否停车
```

---

## 4.2 servo

负责：

```text
输出 50 Hz 舵机 PWM
设置目标角度
限制机械安全角度
返回当前目标角度
返回舵机是否到位
```

不负责：

```text
主动扫描
超声波测距
判断障碍物
控制小车
```

---

## 4.3 obstacle_scanner

负责：

```text
控制舵机看左、中、右
等待舵机稳定
在各方向触发超声波测距
保存左、中、右距离
对多次结果做滤波
给出推荐绕行方向
```

---

## 4.4 obstacle_monitor

负责连续判断：

```text
前方是否存在障碍物
是否需要减速
是否需要停车
障碍物是否已经消失
超声波数据是否失效
```

---

## 4.5 obstacle_avoidance

负责高级避障动作：

```text
停车
扫描
选择方向
请求 CarController 转向
请求保持航向直行
重新寻找线路
返回避障结果
```

它仍然不能直接调用 `Motor_SetSpeed()`。

---

# 5. HC-SR04 非阻塞测距设计

## 5.1 禁止使用长时间阻塞等待

不要采用：

```c
while (ECHO == 0);
while (ECHO == 1);
```

这种写法会阻塞主循环。

因为 ECHO 脉冲可能持续数毫秒，超时甚至可能超过 20 ms，阻塞后会影响：

```text
循迹控制
IMU 更新
按键响应
OLED 更新
任务状态机
电机控制
```

当前工程的控制主线为 20 ms 周期，因此超声波必须使用非阻塞状态机或定时器捕获。

---

## 5.2 推荐测距状态机

```c
typedef enum {
    ULTRASONIC_STATE_IDLE = 0,
    ULTRASONIC_STATE_TRIGGER,
    ULTRASONIC_STATE_WAIT_RISING,
    ULTRASONIC_STATE_WAIT_FALLING,
    ULTRASONIC_STATE_DONE,
    ULTRASONIC_STATE_TIMEOUT
} UltrasonicState;
```

运行流程：

```text
IDLE
→ 产生约 10 μs TRIG
→ WAIT_RISING
→ 捕获 ECHO 上升沿时间
→ WAIT_FALLING
→ 捕获 ECHO 下降沿时间
→ 计算脉宽
→ DONE
```

超过最大等待时间：

```text
WAIT_RISING / WAIT_FALLING
→ TIMEOUT
```

建议使用：

```text
硬件定时器输入捕获
```

次选：

```text
GPIO 上升沿、下降沿中断
+
一个自由运行的微秒计时器
```

不建议完全依赖 20 ms 主循环采样 ECHO，因为 20 ms 分辨率远不足以测量超声波脉宽。

---

## 5.3 距离换算

可使用：

```c
distance_cm = echo_high_time_us / 58.0f;
```

或者：

```c
distance_cm = echo_high_time_us * 0.01715f;
```

建议限制有效范围：

```text
3 cm～250 cm
```

超出范围标记为无效，而不是直接当作“没有障碍物”。

---

## 5.4 测量频率

HC-SR04 不宜过于频繁触发。

建议：

```text
正常前向监测：每 60～100 ms 测量一次
舵机扫描：每个角度测量 3 次
```

不要每 20 ms 都触发一次。

---

# 6. 超声波滤波

单次测量可能受到：

```text
物体表面角度
车体振动
电机电磁干扰
舵机震动
地面反射
障碍物边缘
```

建议每个方向采集 3 次或 5 次，使用中值滤波：

```c
distance_filtered = Median3(d1, d2, d3);
```

同时记录：

```c
typedef struct {
    float distance_cm;
    bool valid;
    uint32_t timestamp_ms;
    uint8_t valid_sample_count;
} UltrasonicMeasurement;
```

不要只保存一个浮点距离。

---

# 7. 舵机控制设计

## 7.1 PWM 参数

常用舵机控制周期：

```text
20 ms
即 50 Hz
```

初始脉宽范围可以设置为：

```text
约 0.7 ms～2.3 ms
```

不要一开始就使用非常极端的脉宽，避免舵机撞击机械限位。

建议定义：

```c
#define SERVO_MIN_ANGLE_DEG       25.0f
#define SERVO_MAX_ANGLE_DEG       155.0f
#define SERVO_CENTER_ANGLE_DEG    90.0f
#define SERVO_LEFT_ANGLE_DEG      145.0f
#define SERVO_RIGHT_ANGLE_DEG     35.0f
```

实际左右方向需要根据安装朝向确认。

---

## 7.2 舵机到位时间

舵机改变角度后不能立即测距。

建议等待：

```text
小角度变化：120～180 ms
大角度变化：200～300 ms
```

第一版统一使用：

```text
250 ms
```

更稳定。

---

## 7.3 正常运行时舵机位置

小车正常循迹时：

```text
舵机固定朝正前方
```

不要让舵机在小车高速运行时不断左右扫动，因为会造成：

```text
正前方检测存在盲区
测距方向不断变化
舵机电流干扰
机械抖动
判断延迟
```

推荐策略：

```text
正常行驶：只测正前方
发现障碍：停车后再左右扫描
```

---

# 8. 障碍物判断逻辑

建议设置三段距离。

```c
typedef struct {
    float slow_distance_cm;
    float stop_distance_cm;
    float clear_distance_cm;
} ObstacleConfig;
```

初始建议：

```text
slow_distance_cm  = 45 cm
stop_distance_cm  = 25 cm
clear_distance_cm = 40 cm
```

其中：

```text
距离 > 45 cm：
正常行驶

25～45 cm：
进入警告状态，可适当降低速度

距离 < 25 cm：
停车

停车后距离重新大于 40 cm：
允许继续
```

使用不同的停止和恢复阈值，可以形成滞回，避免小车在临界距离附近反复启停。

---

## 8.1 连续采样确认

不要因为一次测量小于阈值就立即认定障碍物。

建议：

```text
连续 3 次小于 stop_distance：
确认障碍物

连续 5 次大于 clear_distance：
确认道路恢复
```

状态定义：

```c
typedef enum {
    OBSTACLE_STATE_CLEAR = 0,
    OBSTACLE_STATE_WARNING,
    OBSTACLE_STATE_BLOCKED,
    OBSTACLE_STATE_SCANNING,
    OBSTACLE_STATE_SENSOR_ERROR
} ObstacleState;
```

---

## 8.2 数据超时处理

例如超过 500 ms 没有得到有效测距：

```text
不能直接认为前方安全
```

建议：

```text
低速调试模式：停车并报错
正式比赛模式：根据任务 Profile 决定停车或忽略
```

默认推荐停车，更安全。

---

# 9. 与 20 ms 调度主线的集成

现有顺序为：

```text
1. Imu_Update
2. HeadingObserver
3. Key
4. Menu
5. MissionManager
6. CarController
7. 调试变量
8. OLED
```

建议改为：

```text
1. Imu_Update(0.02f)
2. App_UpdateHeadingObserver()
3. Ultrasonic_Service()
4. Servo_Update_20ms()
5. ObstacleScanner_Update_20ms()
6. ObstacleMonitor_Update_20ms()
7. Key_Update_20ms()
8. Key_GetEvent()
9. Menu_HandleKeyEvent()
10. MissionManager_Update_20ms()
11. ObstacleSafety_Update_20ms()
12. CarController_Update_20ms()
13. 更新调试变量
14. OledUi_Update_20ms()
```

其中真正的 ECHO 脉宽测量仍由：

```text
定时器输入捕获
或 GPIO 边沿中断
```

完成。

20 ms 主循环只负责：

```text
读取是否出现新测量
更新滤波
更新障碍物状态
更新扫描状态机
```

# 10. 安全停车接口

建议在 `CarController` 增加：

```c
void CarController_SetSafetyHold(bool enable);
bool CarController_IsSafetyHoldActive(void);
```

在 `CarController_Update_20ms()` 最后输出电机之前检查：

```c
if (g_safety_hold) {
    Motor_SetSpeed(0, 0);
    return;
}
```

但需要注意：

```text
SafetyHold 只是限制最终电机输出
不能改变当前任务动作
```

因此还应让 `MissionManager` 知道任务当前被障碍物暂停。

建议增加：

```c
void MissionManager_SetExternalHold(bool enable);
bool MissionManager_IsExternallyHeld(void);
```

进入外部 Hold 后：

```text
不增加 mission_elapsed_ms
不增加 action_elapsed_ms
不判断 FOLLOW_DURATION 完成
不切换到下一个动作
保持当前 action_index
```

这样能够避免小车因障碍停车时，任务时间仍然继续累计。

---

# 11. 停车等待避障流程

第一版推荐状态机：

```c
typedef enum {
    OBSTACLE_AVOID_IDLE = 0,
    OBSTACLE_AVOID_DECELERATING,
    OBSTACLE_AVOID_STOPPED,
    OBSTACLE_AVOID_WAIT_CLEAR,
    OBSTACLE_AVOID_RESUMING,
    OBSTACLE_AVOID_FAILED
} ObstacleAvoidState;
```

流程：

```text
正常运行
→ 前方进入 warning_distance
→ 降速
→ 前方进入 stop_distance
→ MissionManager ExternalHold
→ CarController SafetyHold
→ 停车等待
→ 连续确认前方 clear
→ 解除 SafetyHold
→ 解除 ExternalHold
→ 恢复原动作
```

恢复时建议：

```text
FOLLOW_LINE：
清零 PD 的 derivative 历史，避免瞬间大修正

SEEK_LINE：
继续使用原 target_yaw

TURN_90：
不建议自动恢复，进入 ERROR 或重新开始该转弯动作
```

障碍物出现在 90°转弯过程中时，直接暂停后继续转弯可能存在风险。

第一版可以规定：

```text
FOLLOW / SEEK 遇障碍：停车等待后恢复
TURN 遇障碍：停车并进入任务 ERROR
```

# 12. 舵机扫描状态机

建议定义：

```c
typedef enum {
    SCAN_STATE_IDLE = 0,
    SCAN_STATE_MOVE_CENTER,
    SCAN_STATE_WAIT_CENTER,
    SCAN_STATE_MEASURE_CENTER,
    SCAN_STATE_MOVE_LEFT,
    SCAN_STATE_WAIT_LEFT,
    SCAN_STATE_MEASURE_LEFT,
    SCAN_STATE_MOVE_RIGHT,
    SCAN_STATE_WAIT_RIGHT,
    SCAN_STATE_MEASURE_RIGHT,
    SCAN_STATE_RETURN_CENTER,
    SCAN_STATE_COMPLETE,
    SCAN_STATE_FAILED
} ObstacleScanState;
```

扫描顺序建议：

```text
停车
→ 中间确认一次
→ 左侧
→ 右侧
→ 回中
→ 输出扫描结果
```

每个方向：

```text
舵机到位
→ 等待 250 ms
→ 测量 3 次
→ 中值滤波
→ 保存结果
```

扫描结果：

```c
typedef enum {
    OBSTACLE_DIRECTION_NONE = 0,
    OBSTACLE_DIRECTION_LEFT,
    OBSTACLE_DIRECTION_RIGHT,
    OBSTACLE_DIRECTION_BLOCKED,
    OBSTACLE_DIRECTION_UNKNOWN
} ObstacleDirection;
```

推荐判断：

```text
左侧、右侧都小于最小绕行距离：
BLOCKED

左侧比右侧至少大 10 cm：
LEFT

右侧比左侧至少大 10 cm：
RIGHT

两侧差异小于 10 cm：
选择与当前线路结构更匹配的一侧，
或者保持停车等待
```

第一版建议差异不明显时保持停车，不随机选择方向。

# 13. 任务动作接口扩展

## 13.1 增加障碍物处理策略

建议在 `motion_types.h` 增加：

```c
typedef enum {
    OBSTACLE_POLICY_IGNORE = 0,
    OBSTACLE_POLICY_STOP_AND_WAIT,
    OBSTACLE_POLICY_STOP_AND_SCAN,
    OBSTACLE_POLICY_BYPASS_AND_REACQUIRE
} ObstacleHandlingPolicy;
```

然后给 FOLLOW 和 SEEK 动作增加：

```text
obstacle_policy
```

例如：

```c
ACTION_FOLLOW_FOREVER_OBSTACLE(
    OBSTACLE_POLICY_STOP_AND_WAIT,
    timeout
)
```

或者统一宏：

```c
ACTION_FOLLOW_UNTIL_LINE_LOST_WITH_OBSTACLE(
    OBSTACLE_POLICY_STOP_AND_SCAN,
    timeout
)
```

---

## 13.2 不建议新增“随时抢占电机”的模块

不要设计成：

```text
ObstacleManager 发现障碍物
→ 直接 Motor_SetSpeed(0,0)
```

否则会出现：

```text
CarController 在输出速度
ObstacleManager 同时输出 0
两个模块争夺电机
最终行为取决于函数调用顺序
```

正确方式是：

```text
ObstacleMonitor 产生状态
→ MissionManager 暂停
→ CarController 进入 SafetyHold
→ 最终由 CarController 输出 0
```

# 14. 主动绕障方案

## 14.1 当前工程可实现的基础方案

当前工程没有成熟的编码器距离闭环，因此第一版绕障可以采用：

```text
IMU 角度控制
+
按时间直行
+
重新找线
```

流程示例：

```text
1. 发现前方障碍物
2. 停车
3. 舵机扫描左右距离
4. 选择空旷方向
5. 记录 obstacle_start_yaw
6. 向空旷方向转动约 30°～45°
7. 按新航向直行一段时间
8. 调整回原始前进方向
9. 从障碍物侧面经过
10. 向原线路方向转向
11. 进入 SEEK_LINE
12. 找到线路后恢复 FOLLOW_LINE
```

但按时间控制距离会受到：

```text
电池电压
地面摩擦
电机差异
障碍物大小
小车速度
```

影响，因此稳定性有限。

---

## 14.2 推荐的正式绕障方案

正式比赛版建议增加：

```text
编码器距离闭环
IMU 指定角度转向
DRIVE_DISTANCE 动作
TURN_TO_YAW 动作
REACQUIRE_LINE 动作
```

新动作类型：

```c
MOTION_ACTION_TURN_TO_YAW
MOTION_ACTION_DRIVE_HEADING_TIME
MOTION_ACTION_DRIVE_HEADING_DISTANCE
MOTION_ACTION_SCAN_OBSTACLE
MOTION_ACTION_REACQUIRE_LINE
MOTION_ACTION_BYPASS_OBSTACLE
```

推荐绕障子状态机：

```text
BYPASS_STOP
BYPASS_SCAN
BYPASS_SELECT_DIRECTION
BYPASS_TURN_OUT
BYPASS_MOVE_SIDE
BYPASS_TURN_FORWARD
BYPASS_PASS_OBSTACLE
BYPASS_TURN_TO_LINE
BYPASS_REACQUIRE_LINE
BYPASS_COMPLETE
```

# 18. 开发实施顺序

## P0：硬件单模块验证

目标：

```text
HC-SR04 能稳定测距
舵机能稳定旋转
主控不会因舵机动作复位
```

步骤：

```text
1. 确认舵机独立供电和共地。
2. 给 ECHO 增加电阻分压。
3. 完成舵机 35°、90°、145°三位置测试。
4. 完成 HC-SR04 静态距离测试。
5. 检查 10 cm、20 cm、50 cm、100 cm 误差。
6. 检查电机运行时测距是否明显抖动。
```

验收标准：

```text
静止物体距离测量稳定
舵机运动不会导致 MCU 复位
无 ECHO 时能够正常超时
超声波代码不会阻塞主循环
```

# 20. 主要风险

## 风险一：舵机供电导致主控复位

解决：

```text
舵机独立供电
公共 GND
增加大电容
电源线尽量短
```

## 风险二：ECHO 5V 损伤 MCU

解决：

```text
必须使用电阻分压或电平转换
```

## 风险三：阻塞测距破坏循迹

解决：

```text
输入捕获或中断
非阻塞状态机
禁止 while 等待 ECHO
```

## 风险四：超声波误测导致误停车

解决：

```text
中值滤波
连续多次确认
停止和恢复阈值分离
数据有效性检查
```

## 风险五：舵机不断扫描造成正前方盲区

解决：

```text
正常运动时固定朝前
停车后才左右扫描
```

## 风险六：主动绕障后无法重新找到线

解决：

```text
先完成停车和扫描
再开发主动绕障
绕障结束必须进入专门的 REACQUIRE_LINE 状态
设置搜索超时和失败停车
```

# 21. 最终推荐架构

第一阶段稳定版本：

```text
HC-SR04 非阻塞测距
        ↓
ObstacleMonitor
        ↓
发现障碍物
        ↓
MissionManager ExternalHold
        ↓
CarController SafetyHold
        ↓
停车等待
        ↓
障碍消失
        ↓
恢复原动作
```

第二阶段：

```text
发现障碍物
        ↓
停车
        ↓
Servo + ObstacleScanner
        ↓
左中右距离
        ↓
给出推荐绕行方向
```

第三阶段：

```text
扫描完成
        ↓
ObstacleAvoidance
        ↓
TURN_TO_YAW
        ↓
DRIVE_HEADING
        ↓
绕过障碍物
        ↓
REACQUIRE_LINE
        ↓
恢复原任务
```

## 最终优先级

```text
P0  舵机供电和 ECHO 电平保护
P1  HC-SR04 非阻塞测距
P2  前向障碍停车和任务计时冻结
P3  舵机左右扫描与 OLED 观察
P4  IMU 指定角度转向
P5  简单主动绕障和重新找线
P6  编码器距离闭环绕障
```

最关键的开发原则是：

```text
先保证检测到障碍物时能够可靠停车，
再做左右扫描，
最后才做主动绕障。
```

不要一开始就直接实现“扫描—选择方向—绕行—找线”的完整流程，否则超声波、舵机、运动控制和任务恢复四类问题会混在一起，实车排障难度会非常高。
